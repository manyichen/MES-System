# 双星轮胎 MES 数据库优化详细设计 v2

本文档基于以下资料整理：

- `docs/软件分析设计报告.docx`
- `docs/双星轮胎制造MES系统数据库设计书.xlsx`
- `docs/角色权限边界初步说明.md`
- `database/design_output/mes_schema_postgresql.sql`
- 现有后端接口与当前已接入的 PostgreSQL 云数据库

本次优化目标不是推翻原有数据库，而是在原有生产计划、仓储物流、生产报工、质量设备、追溯看板等 36 张业务表基础上，补齐权限管理、组织人员、审计日志、登录会话、审批流、工艺质量标准、设备状态履历等工业管理软件常见设计，使数据库能支撑后续正式权限控制和更完整的 MES 业务闭环。

对应增量脚本：

```text
database/mes_database_optimization_v2.sql
```

该脚本仅做增量增强，使用 `CREATE TABLE IF NOT EXISTS`、`ALTER TABLE ... ADD COLUMN IF NOT EXISTS` 和 `ON CONFLICT DO NOTHING`，不会删除现有表和数据。

## 1. 当前数据库问题

现有数据库能够支撑初版演示流程，但加入登录和权限管理后存在以下不足：

| 问题 | 影响 |
|---|---|
| `mes_user` 只有单个 `role_code` | 无法支撑一个用户多个角色、临时授权、角色权限矩阵 |
| 缺少权限点表 | 无法将“角色”映射到具体接口、模块、按钮、动作 |
| 缺少用户会话表 | 当前登录 token 只能做前端演示，后端无法校验和撤销 |
| 缺少审计日志 | 无法追踪权限变更、报工审核、库存调整、质检判定、维修关闭等关键操作 |
| 缺少权限申请流程 | 报告中的“人事经理发起权限变更、系统维护员处理”无法落库 |
| 缺少组织部门表 | 人员、车间、仓库、质量、设备部门无法统一建模 |
| 工艺管理表不足 | 报告中的 SOP、工序不良原因、轮胎用料参数缺少独立表 |
| 质量标准表不足 | 质检单有了，但质量主管制定标准、分配任务、审核结果的依据不足 |
| 设备状态缺少履历 | 设备状态变化无法追溯“谁在什么时候因为什么改了状态” |
| 通用审批缺失 | 报工、领料、质检、维修、权限申请都需要审批记录 |
| 外部系统/机器人账号边界不清 | 机器人、WMS、财务系统不应混入普通员工账号 |

## 2. 优化原则

1. 保留现有表名和字段，保证当前程序可以继续运行。
2. 新增表优先服务权限、审计和真实业务闭环。
3. 现有 `mes_user.role_code` 保留为兼容字段，后续正式权限以 `mes_user_role` 为准。
4. 业务表只补充必要的人员、审批、更新时间字段，不大规模重构。
5. 机器人、WMS、财务系统作为集成客户端，不作为普通登录用户。
6. 所有高风险操作需要能审计，至少包括：权限变更、密码重置、删除、审批、库存调整、质检判定、维修关闭。
7. 权限控制同时考虑“功能权限”和“数据范围”，例如操作工只能查看本人或本产线相关记录。

## 3. 新增模块总览

| 模块 | 新增表 | 作用 |
|---|---|---|
| 组织人员 | `mes_department` | 部门、车间、仓库、质量、设备、人事等组织结构 |
| RBAC 权限 | `mes_role`, `mes_permission`, `mes_role_permission`, `mes_user_role`, `mes_role_data_scope` | 角色、权限点、角色权限、用户角色、数据范围 |
| 权限流程 | `mes_permission_apply` | 人事经理发起权限变更，系统维护员处理，管理员复核 |
| 登录会话 | `mes_user_session` | 后端可校验、可过期、可撤销的登录状态 |
| 审计日志 | `mes_audit_log` | 关键操作留痕 |
| 系统集成 | `mes_api_client` | 机器人、WMS、财务系统等非人工客户端 |
| 通用审批 | `mes_approval_record` | 报工、领料、质检、维修、权限申请等审批记录 |
| 系统通知 | `mes_notification` | 缺料、质检任务、维修派工、审核结果等通知 |
| 工艺管理 | `mes_sop`, `mes_defect_reason`, `mes_process_parameter` | SOP、不良原因、工艺参数 |
| 质量标准 | `mes_quality_standard`, `mes_quality_standard_item` | 质量主管维护的检验标准和项目 |
| 设备履历 | `mes_equipment_status_log` | 设备状态变化追溯 |

## 4. 权限与用户设计

### 4.1 `mes_user` 增强

保留现有字段：

```text
user_id, username, password_hash, real_name, role_code, department, phone,
enabled, created_at, updated_at, last_login_at
```

新增字段：

| 字段 | 说明 |
|---|---|
| `employee_no` | 员工工号 |
| `dept_id` | 关联 `mes_department.dept_id` |
| `position_name` | 岗位名称 |
| `account_type` | 账号类型：`EMPLOYEE`、`SYSTEM`、`API`、`ROBOT` |
| `password_updated_at` | 密码更新时间 |
| `failed_login_count` | 连续登录失败次数 |
| `locked_until` | 账号锁定截止时间 |
| `created_by` | 创建人 |
| `updated_by` | 更新人 |

兼容策略：

- 当前程序仍可读取 `role_code`。
- 后续正式权限改造时，登录后应读取 `mes_user_role` 的角色集合。
- `role_code` 可作为主角色或演示兼容字段保留。

### 4.2 `mes_department`

用于支撑部门、车间、仓库、质量、设备、人事等组织结构。

关键字段：

| 字段 | 说明 |
|---|---|
| `dept_id` | 部门主键 |
| `dept_code` | 部门编码 |
| `dept_name` | 部门名称 |
| `parent_dept_id` | 上级部门 |
| `dept_type` | 部门类型 |
| `manager_user_id` | 部门负责人 |
| `enabled` | 是否启用 |

建议初始部门：

```text
ADMIN, PMC, WORKSHOP, WAREHOUSE, QUALITY, PROCESS, EQUIPMENT, HR, MANAGEMENT
```

### 4.3 RBAC 表

`mes_role`：角色表。

| 字段 | 说明 |
|---|---|
| `role_code` | 角色编码，例如 `SYSTEM_ADMIN` |
| `role_name` | 角色名称 |
| `role_type` | `SYSTEM`、`BUSINESS`、`MANAGEMENT`、`VIEWER` |
| `role_level` | 角色级别，数值越小权限越高 |
| `data_scope` | `ALL`、`DEPT`、`LINE`、`OWN`、`CUSTOM` |
| `builtin` | 是否内置角色 |
| `enabled` | 是否启用 |

`mes_permission`：权限点表。

权限编码建议采用：

```text
模块.资源.动作
```

例如：

```text
user.read
user.create
user.update_role
planning.manage
production.report
production.report.review
quality.inspect
quality.review
process.manage
equipment.manage
audit.read
```

`mes_role_permission`：角色权限关联表。

`mes_user_role`：用户角色关联表。用于支持一个用户拥有多个角色。

`mes_role_data_scope`：角色数据范围表。用于限制部门、产线、仓库等数据范围。

### 4.4 权限申请

`mes_permission_apply` 支撑报告中的人事经理和系统维护员用例：

```text
人事经理发起权限变更申请
-> 系统维护员处理
-> 系统管理员复核或抽查
-> 写入审计日志
```

建议状态：

| 状态 | 说明 |
|---|---|
| `SUBMITTED` | 已提交 |
| `APPROVED` | 已批准 |
| `REJECTED` | 已驳回 |
| `CANCELLED` | 已取消 |
| `APPLIED` | 已生效 |

## 5. 登录、审计与系统集成

### 5.1 `mes_user_session`

用于正式后端会话：

| 字段 | 说明 |
|---|---|
| `token_hash` | token 哈希，不保存明文 token |
| `issued_at` | 签发时间 |
| `expires_at` | 过期时间 |
| `revoked_at` | 撤销时间 |
| `login_ip` | 登录 IP |
| `user_agent` | 客户端信息 |

后续接口拦截时，前端携带 token，后端查 `mes_user_session` 判断是否有效。

### 5.2 `mes_audit_log`

所有高风险动作写入审计：

| 场景 | 示例 |
|---|---|
| 登录 | 登录成功、登录失败、账号锁定 |
| 用户权限 | 创建用户、禁用用户、修改角色、重置密码 |
| 生产 | 审核报工、驳回报工、删除报工 |
| 仓储 | 库存调整、出库、入库、盘点差异 |
| 质量 | 质检判定、返工创建、质检审核 |
| 设备 | 维修派工、维修关闭、设备状态变更 |
| 系统 | 数据库迁移、接口配置、外部系统同步 |

`before_data` 和 `after_data` 使用 `JSONB` 保存变更前后数据，便于追溯。

### 5.3 `mes_api_client`

用于机器人和外部系统：

| 客户端 | 类型 |
|---|---|
| 运输机器人 | `ROBOT` |
| 质检机器人 | `ROBOT` |
| 生产报工机器人 | `ROBOT` |
| WMS | `WMS` |
| 财务系统 | `FINANCE` |
| ERP | `ERP` |

这些对象不应使用普通员工账号登录，而应使用系统集成凭证。

## 6. 工艺与质量标准优化

### 6.1 工艺管理

报告中的工艺工程师用例包括：

```text
管理 SOP
配置工序不良原因
配置轮胎用料参数
```

新增表：

| 表 | 作用 |
|---|---|
| `mes_sop` | SOP 标准作业指导书 |
| `mes_defect_reason` | 工序不良原因 |
| `mes_process_parameter` | 工艺参数、轮胎用料参数 |

这些表和现有 `mes_process_route`、`mes_product_bom` 互补：

- `mes_process_route` 描述工艺路线和顺序。
- `mes_product_bom` 描述产品用料。
- `mes_sop` 描述操作规范。
- `mes_process_parameter` 描述可量化工艺参数。
- `mes_defect_reason` 为质量判定和返工统计提供标准原因。

### 6.2 质量标准

报告中的质量主管用例包括：

```text
制定质量质检标准
创建抽检工单
分配检验任务
审核检验结果
归档质检报告
```

新增表：

| 表 | 作用 |
|---|---|
| `mes_quality_standard` | 质检标准主表 |
| `mes_quality_standard_item` | 质检项目、标准值、上下限、抽样规则 |

现有 `mes_quality_inspection` 和 `mes_quality_inspection_item` 存储实际检验业务数据；新增标准表存储“应该怎么检”的规则。

## 7. 设备状态与维修闭环

现有设备表已有 `equipment_status`，但缺少状态履历。

新增：

```text
mes_equipment_status_log
```

记录：

| 字段 | 说明 |
|---|---|
| `equipment_id` | 设备 |
| `old_status` | 原状态 |
| `new_status` | 新状态 |
| `changed_by` | 修改人 |
| `changed_reason` | 修改原因 |
| `changed_at` | 修改时间 |

后续设备状态变化应同时更新：

```text
mes_equipment.equipment_status
mes_equipment_status_log
mes_audit_log
```

## 9. 建议接口变更与新增

| `/api/auth/logout` | POST | 撤销当前会话 |

### 9.2 用户、角色、权限

## 10. 实施顺序

建议分三步执行：

1. 先执行 `database/mes_database_optimization_v2.sql`，只新增表和字段，不改业务代码。
2. 后端保留现有登录和业务接口，逐步让登录写入 `mes_user_session`、关键操作写入 `mes_audit_log`。
3. 审核角色权限文档后，再实现接口级权限拦截和角色权限管理页面。

## 11. 与现有功能兼容性

本次设计保持以下兼容：

- `mes_user.role_code` 仍然存在。
- 现有业务表不删除、不重命名。
- 新增字段均允许为空或有默认值。
- 现有前端和后端接口可以继续使用。
- 远程数据库可以通过 `DatabaseMigrationRunner` 执行该增量 SQL。

## 12. 后续仍需确认的问题

1. 是否允许一个用户多个角色，例如“车间管理员 + 设备管理员”。
2. `SYSTEM_MAINTAINER` 是否可以直接改角色，还是必须由 `SYSTEM_ADMIN` 复核。
3. 人事经理是否能查看计件工资明细，还是只能查看工时和报工数量。
4. 管理层是否能查看工资汇总。
5. 工艺标准和质量标准是否需要版本审批流程。
6. 机器人接口是否在本阶段实现，还是只保留数据库设计。
