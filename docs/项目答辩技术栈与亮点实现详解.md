# 双星轮胎 MES 项目答辩技术栈与亮点实现详解

> 依据当前可执行源码整理，版本基线：2026-07-17。重点覆盖 Web 前端、Java 后端、微信小程序、PostgreSQL 数据库、AI 辅助排产、一胎一码追溯和腾讯云服务器部署。
>
> 安全说明：本文只记录配置项名称，不记录项目 `.env` 中的真实数据库密码或 AI API Key。答辩展示配置时也不要把密钥投到大屏上。

## 0. 答辩时先说清楚的结论

### 0.1 一分钟项目概述

本项目是面向轮胎制造现场的 MES，采用 Web 管理端、微信小程序移动端、Java REST 后端和 PostgreSQL 云数据库的多端架构。业务闭环从客户订单、生产任务、BOM 齐套分析、AI 排产建议、制造工单、派工接单、生产报工、质量检验、返工、仓储入库一直延伸到一胎一码公开追溯。

系统的四个主要亮点是：

1. **AI 辅助排产**：先从真实业务库聚合交期、齐套、库存缺口、工艺路线、产线状态、日产能和在制负荷，再调用阿里云百炼大模型生成结构化建议；AI 只辅助决策，PMC 确认后才创建工单。
2. **一胎一码**：只有审核通过且判定合格的轮胎才允许生成独立序列号、随机访问令牌、二维码、标签 PNG 和追溯 PDF；生成过程有数据库事务、数量上限和并发锁。
3. **扫码追溯**：二维码中保存的是带随机令牌的公开追溯 URL。手机或微信识别 URL 后，Web 页面或小程序用令牌查询工单、质检、产品、产线、批次和仓库信息，不把大量业务信息直接写死在二维码里。
4. **云端部署**：腾讯云轻量服务器运行 Nginx、Supervisor 和 Java 21 后端，业务数据通过 PostgreSQL JDBC 连接云数据库；发布脚本支持预编译、配置和追溯文件继承、健康检查及失败回滚。

### 0.2 必须准确使用的技术名词

- 后端**不是 Spring Boot**。它是 `Java 21 + 嵌入式 Tomcat 10.1 + Jersey 3/JAX-RS + JDBC`。
- Web 前端是 `Vue 3 + Vite + Pinia + Vue Router + Lucide`。
- 小程序是微信原生 `JavaScript + WXML + WXSS`，不是 uni-app/Taro。
- 数据库是 PostgreSQL，运行时使用 JDBC 和 `PreparedStatement`，没有使用 MyBatis、Hibernate 或 JPA。
- AI 不是自研神经网络或遗传算法，而是**业务数据约束 + 提示词工程 + 大模型结构化输出 + 后端校验 + 人工确认**。
- 二维码由后端使用 ZXing **本地离线生成**，不是调用在线二维码网站；PDF 由 PDFBox 生成。
- 当前二维码承载的是普通公网 Web URL，不是微信“小程序码”。小程序通过 `wx.scanCode` 也可以解析这个 URL。

### 0.3 已实现、演示配置和待完善边界

| 类别 | 当前事实 |
| --- | --- |
| 已实现 | AI 请求、业务快照、JSON 输出校验、建议预填工单；一胎一码、二维码/标签/PDF、公开接口、小程序扫码；JDBC 云库连接；Nginx/Supervisor 部署和回滚脚本 |

## 2. 总体架构与一次请求怎样流转

### 2.1 生产部署拓扑

```text
浏览器 / 微信小程序
          |
          | HTTP（正式上线应升级 HTTPS）
          v
腾讯云轻量应用服务器 119.45.196.92
  Nginx :80
  ├─ /                 -> /www/wwwroot/mes/frontend/dist（Vue 静态文件）
  └─ /api/*            -> 127.0.0.1:8080（反向代理）
                              |
                              v
                Java 21 + 嵌入式 Tomcat + Jersey
                Controller -> Service -> DAO -> JDBC
                              |             |
                              |             v
                              |       PostgreSQL 云数据库
                              v
                  阿里云百炼 DashScope API

追溯文件：服务器本地 /www/wwwroot/mes/storage/trace（由配置决定）
进程守护：Supervisor；静态与反向代理：Nginx
```

### 2.2 后端分层

后端业务包统一使用：

```text
controller/ -> service/ -> dao/ -> PostgreSQL
                    |
                    └-> entity/、AI 外部服务、追溯文件系统
```

- `Controller/Resource`：通过 JAX-RS 注解声明路径和 HTTP 方法，解析参数，获取当前用户，包装响应，不直接写 SQL。
- `Service`：做必填校验、状态机、角色边界、跨 DAO 编排和异常语义。
- `DAO`：通过 JDBC、`PreparedStatement`、事务和 `ResultSet` 完成数据库访问。
- `entity`：Java bean 或 record，作为请求、响应和数据库行的载体。

分层约束由 `backend/src/test/java/.../ArchitectureLayerTest.java` 自动检查。

### 2.3 Web 请求链

1. Vue 组件调用 `frontend/src/api/http.js` 中的 `api.get/post/put/delete/blob`。
2. `request()` 从 `localStorage` 读取 `mes.session`，自动加 `Authorization: Bearer <token>`。
3. 浏览器请求同源 `/api/...`；本地由 Vite 代理到 `127.0.0.1:8080`，生产由 Nginx 代理。
4. Jersey 扫描 `com.example.messystem` 下的 Resource 和 Provider。
5. `AuthFilter.filter()` 做身份和接口权限校验，`DataScopeFilter` 做产线/仓库数据范围校验。
6. Resource 调用 Service，Service 调用 DAO，DAO 调用 PostgreSQL。
7. 后端使用 `ApiResponse` 返回统一的 `success/message/data` JSON；前端 `request()` 解包 `data`。

### 2.4 小程序请求链

1. 页面通过 `utils/request.js` 的 `get/post/put/download` 请求接口。
2. 请求层从 `utils/session.js` 读取登录会话，按需附加 Bearer Token。
3. `wx.request` 请求 `${getApiBase()}${path}`；追溯公开接口显式传 `{ auth: false }`。
4. 401 时清理会话并 `wx.reLaunch` 回登录页。
5. 服务地址保存在本机 `wx` storage，用户可在登录页或“我的”页修改。

### 3.4 数据库与 SQL

- PostgreSQL 基础设计文件 `database/design_output/mes_schema_postgresql.sql` 包含 37 个基础业务表、46 个索引；后续迁移继续增加权限、会话、数据范围、一胎一码、缺料预警和返工计划等表。
- 广泛使用主外键、唯一约束、索引、`BIGSERIAL`、`RETURNING`、`ON CONFLICT`、CTE、`FOR UPDATE` 和事务。
- 不使用 ORM，SQL 直接位于 DAO，优点是业务 SQL 和事务边界透明；代价是映射代码多、跨数据库迁移成本高。
- 数据库迁移是 SQL 文件 + `DatabaseMigrationRunner` 手工执行，并非 Flyway/Liquibase 自动版本迁移。

## 5. 亮点一：AI 辅助排产

### 5.1 它解决什么问题

传统人工排产要同时查看订单交期、任务数量、物料齐套、库存缺口、产品工艺路线、产线状态、日产能、已有在制工单和剩余负荷。信息分散时容易只看交期而忽略缺料或产能冲突。

本项目的做法不是让模型直接改数据库，而是：

```text
真实数据库硬数据
  -> 后端聚合成受控业务快照
  -> 大模型生成固定 JSON 建议
  -> 后端检查建议完整性和产线 ID
  -> 前端展示依据
  -> PMC 点击“采用建议”
  -> 只预填工单表单
  -> PMC 再次确认并提交
  -> 后端重新校验齐套/产线/工艺后事务落库
```

关键设计思想是“AI 提建议，人做决定，业务规则最终由后端兜底”。

### 5.2 前端怎样触发

#### 入口和权限

- `frontend/src/components/ModuleWorkspace.vue` 的 `showAiPlanning`：只有当前模块是 planning、当前页签是 `workOrders`，且用户拥有 `planning.work_order.create` 时才显示 AI 面板。
- 同文件模板中的 `<AiPlanningPanel @apply="applyAiAdvice" />`：渲染 AI 面板并接收采用事件。
- `frontend/src/config/modules.js` 的 planning 模块 `workOrders` 配置：制造工单页和普通“制定工单”动作。

#### 候选任务

`AiPlanningPanel.vue` 的 `loadCandidates()` 调用：

```http
GET /api/production-tasks
Authorization: Bearer <session-token>
```

前端只保留：

```js
task.taskStatus === 'READY' && task.kittingStatus === 'READY'
```

也就是任务状态和齐套状态都已就绪。这个页面过滤只改善体验，真正的安全校验仍在后端。

#### 生成建议

`AiPlanningPanel.vue` 的 `generateAdvice()` 调用：

```http
POST /api/ai/planning/advice
Content-Type: application/json
Authorization: Bearer <session-token>

{
  "taskIds": [123],
  "horizonDays": 7,
  "objective": "优先满足交期，同时降低缺料和产线冲突风险"
}
```

`AuthorizationPolicy.java` 中 `ai/planning/advice` 规则要求 `planning.work_order.create` 权限。

#### 采用建议

`AiPlanningPanel.applyAdvice()` 不直接创建工单，只 emit：

```js
{
  taskId: 当前任务ID,
  lineId: AI推荐产线ID,
  plannedQty: 当前任务计划数量
}
```

`ModuleWorkspace.applyAiAdvice()` 将这些值合并到“制定工单”动作的 `defaults`，打开通用 `ActionDialog`。PMC 仍要核对产线、数量和批次，再由 `execute()` 调用：

```http
POST /api/work-orders
```

因此“采用建议”只是预填，不是自动落库。

### 5.3 后端调用链

```text
POST /api/ai/planning/advice
  -> AuthFilter + AuthorizationPolicy
  -> AiPlanningAdviceResource.advice()
  -> AiPlanningAdviceService.generate()
      -> AiPlanningDataService.buildSnapshot()
          -> PlanningDao / WorkOrderService -> PostgreSQL
      -> AiPlanningClient.requestAdvice()
          -> DashScope /chat/completions
      -> validateConcreteAdvice()
  -> AiPlanningAdviceResponse
```

快速定位：

| 职责 | 文件/函数 |
| --- | --- |
| API 入口 | `planning/controller/AiPlanningAdviceResource.java`，`advice()` |
| 总编排 | `planning/service/AiPlanningAdviceService.java`，`generate()` |
| 数据快照 | `planning/service/AiPlanningDataService.java`，`buildSnapshot()` |
| 外部调用 | `planning/ai/AiPlanningClient.java`，`requestAdvice()`、`send()` |
| 配置 | `planning/ai/AiPlanningConfig.java` |
| 请求/响应 DTO | `AiPlanningAdviceRequest.java`、`AiPlanningAdviceResponse.java` |
| 权限规则 | `auth/AuthorizationPolicy.java` |
| 前端交互 | `frontend/src/components/AiPlanningPanel.vue` |

### 5.4 业务快照包含什么

`AiPlanningDataService.buildSnapshot()` 组装一个 `Map<String,Object>`，随后由 Jackson 转成 JSON：

| 快照字段 | 数据来源与算法 |
| --- | --- |
| `today` | `LocalDate.now()` |
| `objective` | 用户目标；空值使用默认目标 |
| `horizonDays` | 默认 7，最大 30 |
| `candidateTasks` | 当前选择的一个任务 + 订单号、客户、交期、优先级 |
| `productionLines` | 产线编码、名称、类型、状态、启用标志、日产能 |
| `activeWorkOrderCount` | 按产线统计未完成工单个数 |
| `pendingQty` | 每条产线对所有活动工单求和 `max(0, plannedQty - actualQty)` |
| `schedulable` | 启用且状态不是 `FAULT`/`DISABLED` |
| `selectedProcessRoute` | 当前产品全部工序，按 `processSeq` 升序 |
| `openWorkOrders` | 排除 FINISHED/COMPLETED/CLOSED/CANCELLED 的工单 |
| `kittingAnalyses` | 当前任务的齐套分析 |
| `shortageAlerts` | 当前任务的缺料预警 |
| `materialDemand` | 对每个 BOM 项计算需求、库存和缺口 |

物料计算公式位于 `materialDemand()`：

```text
requiredQty = BOM 单位用量 qtyPerUnit × 任务计划数量 planQty
availableQty = 当前物料可用库存
shortageQty  = max(requiredQty - availableQty, 0)
```

产线负荷不是大模型凭空估计，而是后端先用真实工单算出待完成数量，再交给模型解释和推荐。

### 5.5 为什么必须先齐套

`AiPlanningDataService.selectTasks()` 强制：

- 必须显式选择任务；
- 当前实现一次只能分析一个任务；
- 任务必须真实存在；
- `taskStatus == READY`；
- `kittingStatus == READY`。

齐套分析入口在 `KittingService` 和 `PlanningDao`。它以产品 BOM、任务数量和库存为基础生成 `mes_kitting_analysis` 与 `mes_kitting_shortage_item`。如果短缺，任务进入 `SHORTAGE`，不能进行 AI 工单建议；补料后重新齐套，才进入 `READY`。

### 5.6 怎样调用阿里云百炼

配置来自：

```env
AI_PLANNING_ENABLED=true
DASHSCOPE_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
DASHSCOPE_MODEL=qwen-turbo
DASHSCOPE_API_KEY=<secret>
```

类名和环境变量沿用了 `DASHSCOPE` 命名，模板默认是阿里云百炼公共兼容端点；实际运行可通过 `.env` 覆盖为阿里云其他 OpenAI-compatible 模型服务地址。当前本机配置也是阿里云北京模型服务域名，而不是模板中的公共域名，但 HTTP 协议和下面的调用链不变。真实 Key 不应出现在文档、日志或答辩画面中。

`AiPlanningClient` 使用 JDK 自带的 `HttpClient`：

- 连接超时 8 秒；
- 整个请求超时 45 秒；
- URL：`<baseUrl>/chat/completions`；
- Header：`Authorization: Bearer <DASHSCOPE_API_KEY>`；
- `Content-Type: application/json`；
- `temperature: 0.2`，降低输出随机性；
- 优先要求 `response_format: {"type":"json_object"}`。

发送给百炼的核心请求体等价于：

```json
{
  "model": "qwen-turbo",
  "temperature": 0.2,
  "response_format": { "type": "json_object" },
  "messages": [
    { "role": "system", "content": "轮胎工厂 PMC 排产规则和固定 JSON 字段..." },
    { "role": "user", "content": "排产业务数据如下，请仅输出 JSON：\n<业务快照>" }
  ]
}
```

如果模型接口以 400/422 表示不支持 `response_format`，代码会自动去掉该参数再请求一次。这是兼容不同 OpenAI-compatible 模型的降级路径。

### 5.7 提示词约束

`AiPlanningClient.systemPrompt()` 明确要求模型：

- 身份是轮胎工厂 MES 的 PMC 辅助顾问；
- 只能建议，不能要求自动执行；
- 必须从真实且 `schedulable=true` 的产线中选一条；
- 必须使用实际 lineId、lineCode、lineName；
- 根据数量、日产能和在制负荷给开始/结束时间；
- 判断能否在任务期限和订单交期前完成；
- 不得输出不存在的产线或订单；
- 不得使用“视情况、酌情安排”等空话；
- 只能输出固定字段的合法 JSON。

固定输出字段是：

```json
{
  "orderAssignment": "任务/订单/客户/交期",
  "recommendedLineId": 1,
  "recommendedLine": "产线编码 / 产线名称",
  "recommendedStart": "yyyy-MM-dd HH:mm",
  "recommendedEnd": "yyyy-MM-dd HH:mm",
  "deadlineAssessment": "交期判断及依据",
  "overallAdvice": "明确安排结论",
  "schedulingMethod": "完整工艺执行顺序"
}
```

### 5.8 输出解析与安全兜底

1. 从百炼响应 `choices[0].message.content` 取文本。
2. `stripMarkdownFence()` 兼容模型错误地返回 ```json 代码块。
3. Jackson `readTree()` 强制解析 JSON 对象。
4. `validateConcreteAdvice()` 检查 7 个文字字段非空。
5. `recommendedLineId` 必须能转为 Long，且必须出现在业务快照的产线 ID 集合中。
6. PMC 真正提交工单时，`WorkOrderService.createWorkOrder()` 再检查任务仍为双 READY、产线仍可用、工序属于当前产品。
7. `WorkOrderDao.createWorkOrder()` 在同一事务中：插入工单、将任务从 READY 原子更新为 RELEASED、写操作日志；任何一步失败都回滚。

这说明项目使用了两层约束：大模型输出校验负责防幻觉，工单业务服务负责最终一致性。

### 5.9 AI 功能当前局限，答辩要诚实

- 这是大模型辅助决策，不是可证明全局最优的数学优化器。
- `horizonDays` 和 `objective` 被送入模型，但后端没有自己运行 APS 求解器。
- 后端尚未严格解析和验证 AI 返回的日期格式、起止先后、产能计算公式和交期结论。
- `validLineIds()` 当前检查“真实产线 ID”，没有直接过滤 `schedulable=true`；不过最终创建工单时 `WorkOrderService.requireAvailableLine()` 会拒绝故障/停用产线。
- AI 建议没有单独落库，没有保存 prompt、快照摘要、响应、采用人和是否采用，审计能力可继续增强。
- 外部调用没有通用重试/熔断/限流，仅有 response_format 兼容重试。
- 模型不可用时不会自动切换到本地规则排产，页面会显示错误，生产数据不会被修改。

建议答辩回答：“我们把确定性业务约束放在数据库和 Service，把大模型放在建议层，避免模型直接控制生产。下一步可接入 OR-Tools 做硬约束求解，大模型负责解释方案。”

## 6. 亮点二：二维码生成与一胎一码

### 6.1 两种追溯概念要区分

项目中有两套互补追溯：

| 类型 | 表/代码 | 粒度 | 用途 |
| --- | --- | --- | --- |
| 批次级产品追溯链 | `mes_product_trace`；`dashboard/ProductTrace*` | 订单/任务/工单/批次 | 内部查询订单→任务→工单→报工→质检→返工→质量履历 |
| 单胎二维码追溯 | `mes_tire_instance`、`mes_tire_qrcode`；`trace/*` | 每一条成品轮胎 | 唯一标签、扫码公开查询、产品 PDF、打印记录 |

批次级 `ProductTraceDao.findTraceChain()` 会查询 `order`、`task`、`workOrder`、`product`、`workReports`、`qualityInspections`、`reworkOrders`、`qualityTraces`。一胎一码则只允许从已审核合格质检单生成。

### 6.2 前端怎样发起生成

Web 配置位于 `frontend/src/config/modules.js` 的 trace 模块，搜索 `tire-labels` 可直接定位：

1. 进入“产品追溯 → 轮胎二维码”。
2. `GET /api/tire-labels/generate-options` 加载当前用户可见且 `APPROVED + PASS` 的质检单、仓库和库位。
3. 用户选择质检单、仓库、可选库位和数量。
4. 表单自动由质检单带出 `workOrderId`。
5. 调用：

```http
POST /api/tire-labels/generate
Authorization: Bearer <token>
Content-Type: application/json

{
  "workOrderId": 100,
  "inspectionId": 200,
  "warehouseId": 10,
  "locationId": 20,
  "quantity": 5
}
```

`publicBaseUrl` 请求字段存在，但正常生产应优先使用服务器 `.env` 的 `MES_PUBLIC_URL`，避免客户端随意决定二维码域名。

权限：

- 生成需要 `trace.tire.generate`；
- 记录打印需要 `trace.tire.print`；
- 查看需要 `trace.read` 或 `warehouse.read`；
- Service/Resource 还校验系统管理员或仓库管理员的仓库数据范围。

### 6.3 生成前怎样保证“只有合格轮胎”

`TireTraceService.generate()` 先做基本校验：工单、质检单、仓库、数量都必须有效，单次最多 1000 条。

真正的业务校验在 `TireTraceDao.lockGenerationContext()`：

```sql
from mes_quality_inspection qi
join mes_work_order wo on wo.work_order_id = qi.work_order_id
...
where wo.work_order_id = ? and qi.inspection_id = ?
for update of qi, wo
```

它检查：

- 工单、质检单和仓库真实存在；
- 质检状态必须是 `APPROVED`；
- 判定结果必须是 `PASS`；
- 工单必须关联产品；
- 如果选择库位，库位必须属于目标仓库；
- 合格数量来自已审核报工的 `sum(qualified_qty)`；
- 已生成数量来自 `mes_tire_instance` 的非 VOID 记录。

可生成数量公式：

```text
available = qualifiedQuantity - generatedQuantity
```

如果本次 quantity 超过 available，直接拒绝。`FOR UPDATE` 锁住质检和工单，避免两个并发请求同时读取相同剩余数量后超量生成。

### 6.4 唯一编号怎样生成

`TireTraceService.buildSerialNo()`：

```text
TIRE-yyyyMMdd-六位工单ID-六位顺序号
```

示意：

```text
TIRE-20260717-000364-000019
```

追溯码是：

```text
TRACE-TIRE-20260717-000364-000019
```

表 `mes_tire_instance` 对 `serial_no` 和 `trace_code` 都有 UNIQUE 约束，所以数据库是最终唯一性边界。

### 6.5 二维码里到底存了什么

每条轮胎调用 `nextToken()`：

- `SecureRandom` 生成 18 字节随机数；
- Base64 URL-safe、无 padding 编码；
- 通常得到 24 个字符左右的不可预测 token。

随后构造：

```text
<MES_PUBLIC_URL>/trace-public?token=<随机访问令牌>
```

例如：

```text
https://mes.example.com/trace-public?token=AbCdEf...xyz
```

二维码存的是这个 URL，不是把客户、质检、仓库等全部明文写入码中。好处是：

- 码内容短，识别率更好；
- 页面展示字段可调整，不必重印二维码；
- 可以通过数据库将 `qrcode_status` 设为非 ACTIVE 使旧码失效；
- 避免直接暴露内部数据库主键和完整业务数据。

### 6.6 ZXing 怎样生成二维码

`TraceFileService.generateQrCode()`：

```java
new MultiFormatWriter().encode(
    content,
    BarcodeFormat.QR_CODE,
    800,
    800,
    Map.of(
        EncodeHintType.CHARACTER_SET, "UTF-8",
        EncodeHintType.ERROR_CORRECTION, ErrorCorrectionLevel.H,
        EncodeHintType.MARGIN, 2
    )
)
```

- 尺寸 800×800；
- UTF-8；
- H 级纠错，理论上允许较高比例损伤，适合工业标签；
- 2 个模块的 quiet zone；
- `MatrixToImageWriter.toBufferedImage()` 转成 PNG。

这是后端本地计算，不需要联网，也不把追溯 URL发送给第三方二维码服务。

### 6.7 标签 PNG 和 PDF 怎样生成

`TraceFileService.generate()` 为每条轮胎创建：

```text
storage/trace/<serialNo>/
├─ qrcode.png
├─ label.png
└─ product-info.pdf
```

#### 标签 PNG

`generateLabel()` 使用 Java2D：

- 新建 1200×1700 白色画布；
- 画标题、产品名称、规格和二维码；
- 写入序列号、追溯码、生产工单、批次、质检单、质检结果、仓库和入库时间；
- `ImageIO.write()` 保存 PNG。

#### PDF

`generatePdf()` 使用 PDFBox：

- 新建 A4 `PDPage`；
- `LosslessFactory.createFromImage()` 把标签 PNG 嵌入 PDF；
- 等比缩放并居中；
- 保存 `product-info.pdf`。

#### 文件指纹

`sha256()` 流式读取文件并计算 SHA-256，64 位十六进制结果写入 `mes_trace_document.file_hash`。它用于证明文档版本内容；当前下载路径还没有每次自动复算并拒绝被篡改文件，这是可扩展点。

### 6.8 数据库事务和文件一致性

`TireTraceDao.inTransaction()` 显式：

```text
setAutoCommit(false)
  -> 锁上下文
  -> 插入轮胎实例
  -> 生成三个文件
  -> 插入二维码元数据
  -> 插入标签/PDF元数据和 hash
commit
异常 -> rollback
```

文件系统本身不能参加 JDBC 事务，所以 Service 维护 `generatedFiles` 列表：如果 SQL 或业务步骤失败，回滚数据库并 `deleteQuietly()` 删除已生成文件。这是“数据库事务 + 补偿删除”的最终一致性方案。

### 6.9 相关表

迁移文件：`database/mes_tire_trace_v7.sql`。

| 表 | 关键字段 | 作用 |
| --- | --- | --- |
| `mes_tire_instance` | serial、trace、work_order、inspection、product、warehouse、batch | 每条轮胎实体 |
| `mes_tire_qrcode` | tire_id、access_token、target_url、storage_path、status、print_count | 二维码身份和状态 |
| `mes_trace_document` | type、path、hash、version | 标签/PDF文档元数据 |
| `mes_label_print_task` | tire、printed_by、time、remark | 打印/补打审计 |

### 6.10 文件下载与丢失恢复

内部接口：

```text
GET  /api/tire-labels/{id}/qrcode    image/png
GET  /api/tire-labels/{id}/label     image/png
GET  /api/tire-labels/{id}/document  application/pdf
POST /api/tire-labels/{id}/print     记录打印
```

`TireLabelResource.file()` 使用 `StreamingOutput` 将文件流写入 HTTP 响应，并设置 Content-Type、Content-Length、Content-Disposition 和私有缓存。

如果数据库有二维码/文档记录但磁盘文件丢失，`qrCode()`/`document()` 捕获找不到文件异常，调用同步方法 `repairFiles()`：根据数据库保存的原 targetUrl 重新生成文件并更新路径/hash。这样不会改变已经印在标签上的 URL。

测试 `TraceFileServiceTest` 会真正生成二维码，再用 ZXing 解码并断言内容与 targetUrl 完全相同；还用 PDFBox 打开 PDF 并检查页数。`TireTraceServiceRecoveryTest` 验证文件重建。

### 6.11 当前打印能力的真实边界

系统能够生成/预览标签和 PDF，并记录“谁在何时打印/补打”；当前没有接入 Zebra/TSC 等打印机 SDK，也没有真正下发打印机指令。“记录打印”接口是审计，不等同于硬件打印完成回调。

## 7. 亮点三：扫码与公开追溯

### 7.1 扫码的完整链路

```text
后端生成 URL
  -> ZXing 编码为 QR PNG
  -> 标签打印并贴到单条轮胎
  -> 微信小程序 wx.scanCode 读取二维码文本
  -> tokenFrom() 从 ?token=... 提取令牌
  -> navigateTo('/pages/trace/index?token=...')
  -> GET /api/public/tire-traces/{token}（无需登录）
  -> TireTraceDao.findByToken()
  -> 联表查询轮胎/工单/质检/产品/产线/仓库/库位
  -> 小程序展示产品、质量和入库时间线
  -> 用户可下载并 wx.openDocument 打开 PDF
```

### 7.2 微信小程序怎样扫码

入口：`MES-wechat/pages/scan/index.js`。

```js
wx.scanCode({
  onlyFromCamera: false,
  scanType: ['qrCode'],
  success: result => this.openTrace(result.result, '扫码')
})
```

- `onlyFromCamera: false`：可以相机扫，也可以从相册选择；
- `scanType: ['qrCode']`：只接收二维码；
- `result.result` 是 ZXing 当初编码的 URL 文本。

`tokenFrom(value)` 支持两种输入：

1. 完整 URL：正则提取 `[?&]token=...` 并 `decodeURIComponent`；
2. 不含 `/ ? #` 的纯 token：支持手工输入/粘贴。

无法提取 token 时提示“未识别到有效的产品追溯码”。成功后将最近 8 条记录保存在本机 `mes.trace.history`，再跳转追溯详情页。

### 7.3 小程序怎样查询和打开 PDF

`pages/trace/index.js.loadTrace()`：

```js
get(`/public/tire-traces/${encodeURIComponent(token)}`, { auth: false })
```

`auth:false` 说明公开追溯不使用员工登录 Token。它展示：产品名称/编码/规格、单胎序列、追溯码、生产工单、产线、批次、质检单、合格时间、仓库/库位和入库时间。

`openDocument()` 拼出公开 PDF URL，调用：

```text
wx.downloadFile -> 得到 tempFilePath -> wx.openDocument(showMenu: true)
```

### 7.4 普通手机扫码为什么也能打开

二维码目标是 `/trace-public?token=...`。Web 路由 `frontend/src/router.js` 将 `/trace-public` 标为 `meta.public=true`，不要求员工登录。`PublicTraceView.vue` 再调用同一个公开 API 展示追溯信息和 PDF。

因此：

- 普通相机/微信扫一扫把它当 URL，在浏览器打开 Web 公开页；
- 项目小程序的扫码页把它当文本，提取 token 后打开原生小程序追溯页。

### 7.5 后端公开接口

入口 `trace/controller/PublicTireTraceResource.java`：

```text
GET /api/public/tire-traces/{token}
GET /api/public/tire-traces/{token}/document
```

`AuthFilter.filter()` 对 `path.startsWith("public/tire-traces/")` 直接放行，这是故意设计的客户查询入口。员工内部的二维码列表、生成、预览和打印仍然受认证与权限控制。

`TireTraceDao.findByToken()` 条件是：

```sql
where qr.access_token = ? and qr.qrcode_status = 'ACTIVE'
```

并通过 `SELECT_ITEM` 联接：

```text
mes_tire_instance
  -> mes_work_order
  -> mes_quality_inspection
  -> mes_product
  -> mes_production_line
  -> mes_warehouse
  -> mes_warehouse_location
  -> mes_tire_qrcode
```

`publicView()` 只挑选允许公开的字段，不把用户、内部主键、数据库信息或密码返回客户。

### 7.6 公开令牌的安全模型

- token 由 `SecureRandom` 生成，不使用顺序 ID，难以枚举；
- URL 中只有 token，不暴露 tireId；
- 查询使用 `PreparedStatement`；
- 只有 ACTIVE 令牌有效；
- 文件路径使用 normalize + `startsWith(storageRoot)` 防路径穿越；
- 公开 API 返回字段经过白名单组装。

当前改进点：追溯 access_token 在数据库中是明文（便于等值查询），没有有效期、扫码审计、IP 限流和轮换接口。正式公网可改为 token 哈希索引、增加失效时间/访问日志、Nginx 限流和 WAF。

### 7.7 为什么正式发布小程序需要 HTTPS

当前 `MES-wechat/utils/config.js` 默认是局域网 HTTP API，`project.config.json` 的 `urlCheck:false` 允许开发者工具和真机调试绕过合法域名校验。这是开发/答辩配置，不是微信正式发布配置。

正式发布必须：

1. 给服务器绑定已备案域名；
2. 配置可信 HTTPS 证书；
3. 将小程序默认 API 改为 `https://域名/api`；
4. 在微信公众平台配置 request 和 downloadFile 合法域名；
5. 将 `urlCheck` 恢复正式校验；
6. 将 `MES_PUBLIC_URL` 同步改为 HTTPS 域名后再生成生产标签。

已经打印的二维码 URL 不应随意改域名；若必须迁移，需要旧域名持续做 301/反向代理兼容。

```env
MES_DB_URL=jdbc:postgresql://host:5432/database?sslmode=require
MES_DB_DRIVER=org.postgresql.Driver
```

`DbConfig.value()` 的优先级是：

```text
JVM System Property (-DKEY=value)
  > 操作系统环境变量
  > 当前目录或上一级目录 .env
  > 代码默认值
```

`.env` 被 `.gitignore` 忽略，真实密码不应提交仓库。`DbConfig` 是简单 UTF-8 key=value 解析器，支持注释和引号，不支持完整 shell 表达式。

### 8.2 JDBC URL 怎样生成

`DbConfig.getJdbcUrl()`：

- 如果设置 `MES_DB_URL`，直接使用显式 URL；
- 否则拼为 `jdbc:postgresql://HOST:PORT/DATABASE`。

`getDriverClassName()` 对 PostgreSQL 返回 `org.postgresql.Driver`。

### 8.3 代码怎样建立连接

`common/Db.java`：

```java
Class.forName(DbConfig.getDriverClassName());
return DriverManager.getConnection(
    DbConfig.getJdbcUrl(),
    DbConfig.getUser(),
    DbConfig.getPassword()
);
```

所有 DAO 都通过 `Db.getConnection()` 获取连接，并使用 try-with-resources 自动关闭 `Connection`、`PreparedStatement` 和 `ResultSet`。

普通查询模式：

```java
try (Connection c = Db.getConnection();
     PreparedStatement ps = c.prepareStatement(sql)) {
    ps.setLong(1, id);
    try (ResultSet rs = ps.executeQuery()) { ... }
}
```

事务模式：

```text
connection.setAutoCommit(false)
try -> 多条 SQL -> commit
catch -> rollback -> 继续抛异常
finally -> 恢复 autoCommit/关闭连接
```

项目在工单创建、工单状态流转、登录、轮胎二维码批量生成、打印记录等关键用例中使用事务。

### 8.4 为什么使用 PreparedStatement

- 参数通过 `setString/setLong/setBigDecimal` 绑定，不直接拼接用户输入；
- 降低 SQL 注入风险；
- 正确处理日期、数字和 NULL；
- 让 PostgreSQL 有机会复用执行计划。

少数动态 SQL 只拼接后端固定枚举或固定列名，不接收原始客户端 SQL。

### 8.5 云库网络链路

```text
Java 后端（腾讯云服务器或开发机）
  -> TCP 5432
  -> PostgreSQL 云数据库地址
  -> 数据库账号认证
  -> public schema 中的 mes_* 表
```

云平台侧还需要在数据库白名单/安全组允许应用服务器出口 IP 访问 5432。这是云平台配置，不在 Java 源码中。生产环境不应把 5432 对全网开放，也不应把数据库密码下发给 Vue 或小程序；本项目所有数据库访问都在后端完成。

### 8.6 健康检查

`GET /api/db/ping` 的链路：

```text
DbPingResource -> HealthService -> HealthDao.inspect() -> PostgreSQL
```

返回数据库产品/版本、驱动版本、当前库/用户/服务器时间和关键表是否存在，不返回密码。该接口受登录和 `system.health.read` 权限保护。

部署脚本没有拿管理员 token 调 `/db/ping`，而是向 `/api/auth/login` 提交一个必定错误的探测账号，并要求得到 HTTP 400。能到达数据库并完成账号查询才会返回预期业务错误，因此可作为只读的数据库链路探测。

### 8.7 数据库迁移和初始化

- 基础建表：`database/design_output/mes_schema_postgresql.sql`；
- 共享字段迁移：`mes_shared_field_migration.sql`；
- 权限/数据范围：`mes_permission_implementation_v3.sql`、`mes_data_scope_v4.sql`；
- 一胎一码：`mes_tire_trace_v7.sql`；
- 缺料、返工、账号、管理员等分别有独立增量脚本。

`DatabaseMigrationRunner` 可由 Maven exec 调用并逐条执行 SQL。它不是服务启动时自动跑的 Flyway，所以生产迁移要先备份、明确选择脚本、人工验收。其简单分号切分器适合当前普通 DDL/DML，不适合包含复杂 PL/pgSQL 函数体的脚本。

### 8.8 当前连接方案的局限

- 使用 `DriverManager` 每次新建物理连接，没有 HikariCP 等连接池；并发量大时连接成本高。
- 代码默认 URL 没有显式 SSL 参数；可通过 `MES_DB_URL` 增加 `sslmode=require/verify-full`。
- 没有统一查询超时、连接超时和慢 SQL 监控。
- `HealthService` 会向有权限的管理员显示 host/user；不会显示密码，但正式生产仍可进一步脱敏。
- 远程数据库写测试默认通过 `RemoteDatabaseTestGuard` 跳过，防止单元测试污染云库，这是正确的安全边界。

## 9. 服务器部署：怎么构建、怎么运行、怎么回滚

### 9.1 前端怎样生成

```powershell
cd MES/frontend
npm.cmd install
npm.cmd test
npm.cmd run build
```

Vite 读取 `frontend/src`，清空并生成 `frontend/dist`：

```text
frontend/dist/
├─ index.html
└─ assets/index-<content-hash>.js/css/svg
```

文件名带内容 hash，Nginx 可安全设置静态资源缓存。Vue Router 使用 history 模式，所以 Nginx 的 `try_files $uri $uri/ /index.html` 和后端 StaticFileServlet 都提供 SPA 回退。

### 9.2 后端怎样构建

验证和打 WAR：

```powershell
cd MES
.\.tools\apache-maven-3.9.11\bin\mvn.cmd -pl backend test package
```

`backend/pom.xml` 的 `maven-war-plugin` 把 `frontend/dist` 作为 Web 静态资源打入 WAR。

当前服务器实际运行：

```bash
scripts/run-backend.sh
  -> 读取根目录 .env 并 export
  -> cd backend
  -> mvn -DskipTests compile exec:java
  -> com.example.messystem.MesBackendApplication
```

`MesBackendApplication.main()`：

1. 读取 host/port，生产是 `127.0.0.1:8080`；
2. 查找 `frontend/dist`；
3. 创建嵌入式 Tomcat；
4. Jersey Servlet 映射 `/api/*`；
5. StaticFileServlet 映射 `/`；
6. 设置 MIME、静态文件和 history fallback；
7. 启动并 `await()` 常驻。

### 9.3 发布包怎样生成

仓库已有 `release/mes-web-v1-20260717.tar.gz` 和 `SHA256SUMS.txt`，包内含：

- Maven POM；
- 后端主源码和资源；
- 已构建的 `frontend/dist`；
- `scripts/run-backend.sh`；
- Nginx、Supervisor、安装脚本和生产 env 示例。

现有仓库**没有一份固定的“生成 tar.gz”脚本**，因此不要声称发布包生成已经全自动化。当前可复现流程应是先构建前端/测试后端，再用 tar 选择上述路径并计算 SHA-256；正式项目应补一个 `build-release.ps1/sh` 固化文件清单。

仓库内当前包的 SHA-256 与 `release/SHA256SUMS.txt` 一致，安装前服务器使用：

```bash
sha256sum -c SHA256SUMS.txt
```

校验传输是否损坏或被替换。

### 9.4 Nginx 做什么

`deploy/nginx/mes.conf`：

```text
listen 80
server_name 119.45.196.92
root /www/wwwroot/mes/frontend/dist

/api/* -> proxy_pass http://127.0.0.1:8080
/       -> Vue 静态资源与 index.html fallback
```

它把公网入口统一为 80，后端 8080 只绑定环回地址，减少暴露面。反向代理传递 Host、真实 IP、转发链和协议，并给 AI 等较慢请求配置较长 read timeout。

### 9.5 Supervisor 做什么

`deploy/supervisor/mes-backend.conf`：

- 工作目录 `/www/wwwroot/mes/backend`；
- 命令调用 `scripts/run-backend.sh`；
- `autostart=true`、`autorestart=true`；
- 指定 Java 21 的 `JAVA_HOME`；
- stdout/stderr 写入 `/www/wwwlogs`；
- 停止时杀掉进程组，避免 Maven/Java 子进程残留。

当前配置用 root 运行，简化了部署；正式生产可创建最小权限 `mes` 用户并限制目录权限。

### 9.6 安装脚本怎样做到失败回滚

`deploy/install-release.sh` 的关键步骤：

1. 检查 root、压缩包和 `tar/java/mvn/curl/nginx/supervisorctl/systemctl`。
2. 解压到带时间戳的 stage 目录，不立即覆盖线上目录。
3. 检查 POM、后端入口、`frontend/dist/index.html`、启动脚本和运维配置是否齐全。
4. 从当前版本或旧目录继承 `.env`，权限设为 600；复制 `storage`，保证历史二维码文件不丢。
5. 在 stage 中先 `mvn -DskipTests compile`，编译失败不会动线上版本。
6. 备份旧应用、Nginx 和 Supervisor 配置。
7. 原子移动 stage 为正式目录，安装配置并 reload/restart。
8. 最多等待 45 秒探测后端首页。
9. 用 Host Header 探测 Nginx、检查 Supervisor RUNNING。
10. 用假账号登录探测数据库链路，预期 HTTP 400。
11. 任一步失败触发 `trap restore_previous_release ERR`，停止新进程、移走失败版本、恢复旧应用和配置并重启。

这是本项目部署中的重要工程亮点：不是直接覆盖线上目录，而是“先验证、再切换、失败恢复”。

### 9.7 GitHub Actions 的边界

`.github/workflows/deploy.yml` 还存在一套 main push 后 zip、scp、SSH 执行 `/www/wwwroot/deploy-mes.sh` 的流程。它引用的服务器脚本不在当前仓库中，并使用 `sshpass` 密码登录；而当前文档和发布物主要采用 `install-release.sh + tar.gz + SHA256`。

答辩时可说“仓库有 CI/CD 原型，但当前可审计、可回滚的正式交付流程以 install-release.sh 为准”。后续应统一两条部署路径，改用 SSH Key/OIDC、制品仓库和审批环境。

### 9.8 当前生产部署还缺什么

- `mes.conf` 只有 HTTP 80，没有 HTTPS 443 和证书；
- 没有域名，二维码写的是 IP URL；
- Java 每次启动都由 Maven 编译，启动慢且依赖 Maven/依赖缓存；更成熟方案应运行预构建 JAR/WAR；
- 追溯文件存在单机磁盘，服务器损坏需要备份恢复；可迁移到对象存储并保留不可变版本；
- 没有容器化、蓝绿流量切换或集中日志；当前 stage/backup 已实现目录级近似蓝绿和回滚；
- 防火墙和云数据库白名单是云平台配置，需要单独留存运维证据。

## 10. 认证、权限与数据安全

### 10.1 登录和密码

- `PasswordHasher` 使用 `PBKDF2WithHmacSHA256`；
- 每个密码随机 16 字节盐；
- 120,000 次迭代；
- 256 位派生密钥；
- 比较时使用常量时间逻辑；
- 连续 5 次密码错误锁定 15 分钟。

### 10.2 会话 Token

- `TokenHasher.newToken()` 用 `SecureRandom` 生成 32 字节随机 Token；
- 客户端拿到原 Token；
- 数据库只保存 SHA-256 `token_hash`；
- 默认 8 小时过期；
- 退出登录写 `revoked_at`；
- 角色调整等操作可撤销旧会话。

注意：员工会话 Token 只存 hash；二维码公开访问 token 当前是另一套机制，表中保存明文用于查询，二者不要混淆。

### 10.3 RBAC 和默认拒绝

`AuthFilter`：

1. 解析 `Authorization: Bearer`；
2. 查询有效 session、用户、角色、权限和数据范围；
3. `AuthorizationPolicy.requiredPermissions(method,path)` 匹配接口；
4. 没有配置规则的接口默认 403；
5. 有任一所需权限才放行。

前端隐藏菜单/按钮只是体验，后端权限过滤器才是安全边界。

### 10.4 行级数据范围

`DataScopeFilter` 和 `DataScopeService` 按当前用户的 `lineIds`、`warehouseIds`：

- 在请求路径带具体 ID 时提前校验；
- 在列表响应时过滤不属于授权产线/仓库的数据；
- 对工单、任务、订单、报工、质检、设备、仓库、库存、追溯等跨表计算可见 ID。

一胎一码生成还在 Resource 中额外限制仓库角色和目标仓库范围。

## 13. 高频答辩问题与建议回答

### Q1：AI 排产是不是把订单直接发给 ChatGPT？

不是。后端只针对一个已齐套任务构造受控快照，包含排产必要的订单、交期、数量、产线、负荷、工艺和物料信息，通过阿里云百炼 OpenAI-compatible 接口请求结构化建议。真实数据库密码、用户密码和 AI Key 不进入请求。AI 不直接写库。

### Q2：为什么叫 AI 辅助排产，而不是自动排产？

生产排程涉及设备、人员和现场异常，模型输出不能作为最终事实。系统让 AI 做信息综合和方案解释，PMC 点击采用后只预填工单，最后由 `WorkOrderService` 重新检查齐套、产线和工艺并人工提交。这是 human-in-the-loop。

### Q3：AI 如何避免胡编一条不存在的产线？

提示词只允许从业务快照选真实产线；返回后后端检查 `recommendedLineId` 是否在真实产线 ID 集合；最终创建工单又检查产线是否启用且不是故障/停用。三层约束减少幻觉影响。

### Q4：AI 断网或 API Key 失效会不会影响生产数据？

不会修改数据。`AI_PLANNING_ENABLED=false`、Key 缺失、超时或非 2xx 都返回业务错误，前端显示失败，工单不会创建。PMC 仍可使用普通“制定工单”流程。

### Q5：齐套分析是什么？

对任务产品的 BOM 逐项计算 `单耗×计划数量`，再比较可用库存。没有缺口则任务和齐套状态进入 READY；有缺口则写缺料明细和预警，任务不能制定工单。它是 AI 排产之前的硬约束。

### Q6：二维码怎么生成？是不是第三方 API？

不是。后端用 ZXing 的 `MultiFormatWriter.encode()` 把追溯 URL编码成 800×800、UTF-8、H 级纠错二维码。Java2D 将二维码和轮胎信息组合成标签，PDFBox 转成 A4 PDF，全程可离线完成。

### Q7：二维码中为什么不直接存所有轮胎信息？

码只存随机 token 的 URL。业务详情由服务器实时查询。这样二维码短、识别更稳定，展示内容可以升级，令牌也可以在数据库中停用，同时不暴露数据库 ID 和过多内部数据。

### Q8：一胎一码如何保证不多生成？

数据库查询合格报工数量和已生成数量，计算剩余额度；事务中使用 `FOR UPDATE` 锁住对应质检单和工单；编号还有唯一约束。并发请求无法同时超领同一批合格数量。

### Q9：数据库成功但文件生成失败怎么办？

文件生成处于数据库事务编排中，异常会触发 rollback；如果文件已部分生成，Service 记录路径并补偿删除。反过来历史文件丢失时，可根据数据库中不变的 targetUrl 自动重建。

### Q10：扫码为什么不用登录？安全吗？

这是面向客户/现场的公开追溯场景，公开接口只返回白名单字段。访问凭证是 SecureRandom 生成的不可预测 token，不是顺序 tireId，并且只接受 ACTIVE 状态。正式公网还应加限流、访问日志和 token 生命周期。

### Q11：这是二维码还是小程序码？

是标准 QR Code，内容是 Web URL。任何扫码工具可打开 Web 追溯页；项目小程序用 `wx.scanCode` 读出同一 URL 并提取 token，在原生页面展示。它不是通过微信接口生成的小程序码。

### Q12：云数据库怎么连？前端会不会拿到密码？

只有 Java 后端读取服务器 `.env`，用 PostgreSQL JDBC 连接。Vue 和小程序只访问 REST API，永远不拿数据库账号密码。所有 SQL 参数通过 `PreparedStatement` 绑定。

### Q13：为什么没有使用 Spring Boot/MyBatis？

项目选择了更轻量、透明的嵌入式 Tomcat + Jersey/JAX-RS + JDBC。优点是 REST 生命周期、SQL和事务边界清晰，适合展示底层原理；缺点是连接池、事务模板和映射要自己管理。规模继续扩大时可引入 HikariCP 和成熟迁移工具。

### Q14：前端和后端怎么部署在一起？

Vite 先生成 `frontend/dist`。Nginx直接服务该目录，将 `/api/*` 反代到本机 8080 Java 后端。Java 自己也能托管 dist，便于本地一体运行；生产入口主要由 Nginx 承担。

### Q15：服务器进程挂了怎么办？

Supervisor 设置 autostart/autorestart。发布脚本先在 stage 编译，再备份和切换；如果 Nginx、Supervisor、HTTP 或数据库探测失败，ERR trap 自动恢复旧目录和旧配置。

### Q16：为什么小程序现在关闭域名校验？

这是局域网真机联调配置，因为手机不能用开发机的 127.0.0.1。正式发布前必须换 HTTPS 域名并在微信公众平台登记合法域名，当前配置不能直接当生产安全配置。

### Q17：数据库表很多，怎样保证关联正确？

DDL 使用主外键和唯一约束；Service 做状态和关联校验；DAO 写事务。例如创建批次追溯前检查订单→任务→工单链和产品一致，生成轮胎码前检查工单→质检→报工→仓库链。

### Q18：你们怎样测试亮点？

后端 JUnit 覆盖架构分层、权限、齐套/工单、产品追溯、二维码可解码、PDF可读取和文件恢复；Web 有表单/显示/工艺和 Playwright 冒烟；小程序有权限、操作配置和后端路径契约；Postman/Newman 生成 API 回归报告。

## 14. 建议的现场演示顺序

### 14.1 AI 排产（3 分钟）

1. 用 PMC 账号登录。
2. 打开“计划与工单 → 生产任务”，展示某任务已经齐套；必要时先演示齐套分析。
3. 打开“制造工单”，指出 AI 面板只有有权限的 PMC 可见。
4. 选择任务、周期和目标，点击“生成工单决策建议”。
5. 解释推荐产线、开始/结束、交期判断和工艺路线来自真实快照。
6. 点击“采用建议制定工单”，强调这里只预填。
7. 检查批次后提交，再展示任务变 RELEASED 和工单日志。

备用方案：百炼网络不可用时，不要反复等待。展示已有截图/结果并直接打开 `AiPlanningClient.systemPrompt()`、`AiPlanningDataService.buildSnapshot()` 和 `WorkOrderService.createWorkOrder()` 讲代码链。

### 14.2 一胎一码（3 分钟）

1. 准备一条审核通过且 PASS、仍有剩余合格数量的质检单。
2. “产品追溯 → 轮胎二维码 → 生成轮胎标签”。
3. 选择质检单、仓库、库位和数量 1，生成。
4. 查看二维码和标签，打开 PDF。
5. 用微信小程序扫码，展示产品、工单、批次、质检和仓库时间线。
6. 回到 Web 记录打印，展示 printCount 增加。

不要在答辩前对同一质检单生成完全部剩余数量，否则现场会被“已全部生成”规则拒绝。

### 14.3 云端与部署（1 分钟）

1. 展示浏览器公网入口。
2. 打开 Nginx 配置说明 `/` 与 `/api` 分流。
3. 打开 Supervisor 配置说明进程守护。
4. 打开 `install-release.sh` 的 stage、backup、health probe、rollback。
5. 登录管理员，从系统维护页查看数据库健康信息；不要直接展示 `.env`。

## 16. 可继续改进的技术路线

按答辩价值和工程优先级排序：

1. 配置域名和 HTTPS，完成小程序合法域名与正式发布。
2. 引入 HikariCP、数据库 SSL、查询超时、慢 SQL 与连接指标。
3. 用 Flyway/Liquibase 管理迁移版本和校验和。
4. AI 建议写审计表：快照 hash、模型、prompt 版本、响应、采用人、采用结果。
5. 用 OR-Tools/CP-SAT 求解硬约束排程，大模型负责解释和风险摘要。
6. 对 AI 日期、产能、可排产状态做确定性二次计算，而不只校验字段和 ID。
7. 给公开追溯加限流、访问日志、token hash/过期/轮换和防批量爬取。
8. 将追溯文件迁移到带版本和生命周期的对象存储，同时保留 SHA-256 校验。
9. 接入真实标签打印机任务队列和打印状态回调。
10. 生成预构建可执行制品，服务器不再每次用 Maven 编译；统一 GitHub Actions 与回滚安装流程。

## 17. 最后一段答辩总结

本项目的价值不只是把 MES 功能做成页面，而是把制造业务规则落实到了可验证的代码链中：AI 排产使用真实业务快照且不越过人工决策，一胎一码只从合格数量生成且用事务防止超量，扫码通过随机令牌安全地连接线上追溯数据，云端部署则用同源反向代理、进程守护、配置隔离、完整性校验和失败回滚保证系统可运行。当前系统也明确保留了工程边界：AI 不是确定性最优求解器，小程序生产发布还需要 HTTPS，数据库连接和公开追溯仍有可扩展空间。能够准确讲清“已经实现什么、为什么这样设计、最终由哪层兜底、下一步如何演进”，比把所有能力笼统称为智能化更有说服力。
