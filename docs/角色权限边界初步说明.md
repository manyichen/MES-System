# MES 系统角色权限边界初步说明

本文档根据《软件分析设计报告》中的业务用例图、系统用例图、活动图说明，以及当前系统已经实现的模块进行修订。报告中的用例图由初学阶段绘制，本文档在保留核心业务角色的基础上，对部分不适合直接落库的角色进行了工程化调整，作为后续正式 RBAC 权限设计的审核稿。

当前系统已经实现：用户表、角色字段、管理员账号、登录接口、简易登录界面。接口级权限拦截暂不启用，待本文档审核确认后再落地。

## 1. 用户表与登录基础

用户表：`mes_user`

远程数据库当前主键：`user_id`

核心字段：

| 字段 | 说明 |
|---|---|
| `user_id` | 用户主键 |
| `username` | 登录用户名，唯一 |
| `password_hash` | 登录密码哈希，不保存明文密码 |
| `real_name` | 用户真实姓名 |
| `role_code` | MES 角色编码，用于权限管理 |
| `department` | 所属部门 |
| `phone` | 联系电话 |
| `enabled` | 是否启用，`1` 启用，`0` 禁用 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |
| `last_login_at` | 最近登录时间 |

系统管理员初始化账号：

| 项 | 值 |
|---|---|
| 用户名 | `admin` |
| 初始密码 | `123456`（仅限开发与验收环境） |
| 角色 | `SYSTEM_ADMIN` |
| 权限定位 | 最高权限，拥有系统全部操作权 |

## 2. 报告角色与工程化调整

《软件分析设计报告》中出现的主要参与者包括：

| 报告角色 | 本文档处理方式 | 说明 |
|---|---|---|
| PMC 计划员 | 保留为 `PMC_PLANNER` | 负责订单、计划、齐套、工单 |
| 车间管理员 | 保留为 `WORKSHOP_MANAGER` | 比“生产主管”更贴合报告表述 |
| 生产操作工 | 保留为 `PRODUCTION_OPERATOR` | 一线执行和报工 |
| 仓库管理员 | 保留为 `WAREHOUSE_ADMIN` | 报告中负责领料、备料、库存核对 |
| 质检员 | 保留为 `QUALITY_INSPECTOR` | 执行检验、录入结果 |
| 质量主管 | 新增为 `QUALITY_MANAGER` | 制定标准、创建抽检任务、审核质检 |
| 工艺工程师 | 新增为 `PROCESS_ENGINEER` | 维护 SOP、工序、不良原因、用料参数 |
| 设备管理员 | 新增为 `EQUIPMENT_ADMIN` | 设备台账、状态、故障接收、维修派发 |
| 设备维护员 | 保留为 `EQUIPMENT_MAINTAINER` | 接收维修任务、维修、提交结果 |
| 人事经理 | 新增为 `HR_MANAGER` | 员工信息维护、权限变更申请 |
| 系统维护员 | 新增为 `SYSTEM_MAINTAINER` | 处理权限变更、系统运维 |
| 总经理/企业管理层 | 调整为 `GENERAL_MANAGER` | 经营看板、统计查询、决策反馈 |
| 运输机器人、质检机器人、生产报工机器人 | 不作为登录用户 | 作为设备/系统集成对象，不分配人工登录角色 |
| WMS、财务系统、生产管理系统 | 不作为登录用户 | 作为外部系统或内部系统边界，不进入 `mes_user` 人员角色 |

说明：机器人和外部系统在用例图中可以作为参与者，但不建议直接放入人员权限表。后续如需接口对接，应使用“系统账号/API 凭证”单独管理，不与员工角色混用。

## 4. 权限边界原则

1. 系统管理员拥有最高权限，但关键操作仍应记录审计日志。
2. 系统维护员可以处理权限变更和系统运维，但不默认拥有全部业务数据修改权。
3. 人事经理维护员工基础信息，可以发起权限变更申请，但不直接给自己或他人最终授权。
4. 计划、车间、仓库、质量、工艺、设备各模块按职责分离，不能因为能查看工单就能修改所有工单。
5. 审批权和执行权分离。例如：操作工提交报工，车间管理员审核；质检员录入结果，质量主管审核。
6. 机器人、WMS、财务系统等作为外部或自动化参与者，不使用普通员工账号登录。
7. 管理层以看板、报表、追溯、反馈为主，原则上不直接录入一线业务数据。
8. 删除、高风险库存调整、权限变更、密码重置、审批驳回等操作必须做审计记录。

## 5. SYSTEM_ADMIN 系统管理员

可以做：

| 模块 | 权限 |
|---|---|
| 用户与权限 | 创建用户、禁用用户、重置密码、修改角色、查看全部用户 |
| 系统维护 | 执行数据库迁移、系统配置、接口配置、部署配置 |
| 计划工单 | 全部查询、新增、修改、删除、状态调整 |
| 仓储物流 | 全部物料、仓库、库存、领料、拣货、配送操作 |
| 生产报工 | 全部报工新增、修改、审核、删除、工资结果查看 |
| 质量管理 | 全部质检、返工、追溯、质量判定操作 |
| 工艺管理 | 全部 SOP、工序、不良原因、用料参数维护 |
| 设备维护 | 全部设备、报修、维修派工、维修验收、状态调整 |
| 看板反馈 | 全部看板、报表、反馈处理 |

不能做：

| 边界 | 说明 |
|---|---|
| 查看明文密码 | 数据库只保存哈希，任何角色都不应查看明文密码 |
| 绕过审计 | 后续正式实现中，管理员关键操作也必须留痕 |

## 6. SYSTEM_MAINTAINER 系统维护员

对应报告用例：接收权限变更申请、变更人员权限、系统运维。

可以做：

| 模块 | 权限 |
|---|---|
| 用户与权限 | 查看用户账号状态，处理人事经理提交的权限变更申请 |
| 权限配置 | 修改用户角色、启用/禁用账号、重置密码 |
| 系统运行 | 查看系统健康状态、数据库连接状态、接口错误 |
| 审计日志 | 查看登录、权限变更、关键操作日志 |

不能做：

| 禁止项 | 说明 |
|---|---|
| 直接业务审批 | 不直接审核报工、质检、维修、领料等业务 |
| 随意修改业务数据 | 不修改生产、质量、库存、设备台账等业务记录 |
| 修改自己的最高权限 | 自身权限变更应由系统管理员或双人复核 |

## 7. HR_MANAGER 人事经理

对应报告用例：维护员工信息、发送权限变更申请、查看员工报工单。

可以做：

| 模块 | 权限 |
|---|---|
| 员工信息 | 新增、维护员工基础信息：姓名、部门、电话、岗位状态 |
| 权限申请 | 发起用户启用、禁用、角色变更、调岗权限申请 |
| 报工查询 | 查看员工报工单、工时、出勤相关基础数据 |
| 统计查询 | 查看人员维度的工时、报工、岗位分布统计 |

不能做：

| 禁止项 | 说明 |
|---|---|
| 最终授权 | 不能直接修改 `role_code`，只能提交申请 |
| 生产业务 | 不能创建工单、审核报工、修改计划 |
| 质量业务 | 不能录入或审核质检结果 |
| 仓储业务 | 不能修改库存和领料状态 |
| 设备业务 | 不能派发或关闭维修工单 |

## 8. GENERAL_MANAGER 总经理/企业管理层

对应报告用例：查看设备信息、轮胎产量、订单完成进度、综合生产看板。

可以做：

| 模块 | 权限 |
|---|---|
| 首页看板 | 查看订单、产量、质量、设备、库存等汇总指标 |
| 计划工单 | 查看订单完成进度、任务状态、延期风险 |
| 生产报工 | 查看产量、工时、报工汇总、计件工资汇总 |
| 质量管理 | 查看合格率、不良率、返工率、质量追溯统计 |
| 设备维护 | 查看设备状态、故障次数、维修及时率 |
| 仓储物流 | 查看库存风险、缺料预警、配送效率 |
| 管理反馈 | 查看并处理管理层反馈、督办异常闭环 |

不能做：

| 禁止项 | 说明 |
|---|---|
| 一线录入 | 不直接提交报工、质检、领料、维修记录 |
| 用户授权 | 默认不能创建用户或修改角色 |
| 删除数据 | 不允许删除业务记录 |
| 修改基础台账 | 不直接修改库存、设备、工艺、物料等基础数据 |

| 返工协同 | 接收质量返工单，重新安排返工生产任务 |
| 看板查询 | 查看计划完成率、订单进度、缺料风险 |

不能做：

| 禁止项 | 说明 |

| 一线报工 | 不能代替操作工提交报工 |

对应报告用例：查看与接收生产工单、跟踪生产进度、查看生产看板、派发制造工单、审核报工、审核领料需求、派发维修工单或参与派发。

可以做：

| 模块 | 权限 |
|---|---|
| 生产执行 | 接收生产工单、派发制造任务、查看工单详情 |
| 进度管理 | 跟踪生产进度、查看生产看板、处理生产异常 |
| 报工管理 | 审核报工单、驳回异常报工、查看计件汇总 |
| 领料协同 | 审核操作工领料申请、下发领料任务至仓库 |
| 设备异常 | 接收设备异常上报，确认是否需要维修派工 |
| 维修协同 | 查看维修进度，审核维修结果是否满足生产恢复要求 |

不能做：

| 禁止项 | 说明 |
|---|---|
| 用户权限 | 不能管理用户角色 |
| 计划源头 | 不能越权创建客户订单或修改销售订单 |
| 库存台账 | 不能直接修改库存数量 |
| 质检最终结论 | 不能替代质量主管做质量放行 |
| 工艺标准 | 不能维护 SOP 和工艺参数 |

## 11. PRODUCTION_OPERATOR 生产操作工

对应报告用例：接收制造工单、创建领料任务、生产报工、上报设备故障、查看输送线状态、接收物料、查看工资明细。

可以做：

| 模块 | 权限 |
|---|---|
| 工单执行 | 查看并接收分配给本人、本工位或本产线的制造工单 |
| 领料申请 | 创建领料任务，确认接收物料 |
| 生产报工 | 提交报工数量、合格数量、不合格数量、工时、批次 |
| 设备异常 | 上报设备故障、补充异常信息 |
| 结果查看 | 查看本人报工审核结果、本人相关工资明细 |
| 状态查看 | 查看输送线、机器人配送、工单执行状态 |

不能做：

| 禁止项 | 说明 |
|---|---|
| 用户信息 | 不能查看用户列表、不能修改用户权限 |
| 计划创建 | 不能创建订单、生产任务、生产工单 |
| 报工审核 | 不能审核自己的报工 |
| 计件工资 | 不能修改计件单价和工资结果 |
| 仓储台账 | 不能修改库存、库位和仓库基础数据 |
| 质检结论 | 不能录入或审核质检结论 |
| 设备台账 | 不能修改设备状态，只能上报故障 |

## 12. WAREHOUSE_ADMIN 仓库管理员

对应报告用例：接收领料信息、核对物料、上传备料存放明细、推送备料异常、配合 WMS 库存扣减。

可以做：

| 模块 | 权限 |
|---|---|
| 仓储基础 | 维护物料、仓库、库位、基础库存信息 |
| 领料处理 | 接收领料任务、核对库存、备料、拣货、确认发料 |
| 配送协同 | 创建或确认配送任务，跟踪运输机器人状态 |
| 库存流水 | 登记出入库流水，查看库存扣减结果 |
| 异常反馈 | 推送缺料、物料不符、配送异常、退料异常 |

不能做：

| 禁止项 | 说明 |
|---|---|
| 用户权限 | 不能管理用户角色 |
| 生产报工 | 不能提交或审核生产报工 |
| 生产计划 | 不能创建或调整生产任务 |
| 质检结果 | 不能录入或审核质检结果 |

## 13. QUALITY_MANAGER 质量主管

对应报告用例：查看产品信息、创建抽检工单、制定质检标准、分配检验任务、查看并审核检验结果、归档质检报告。

可以做：

| 模块 | 权限 |
|---|---|
| 质检计划 | 根据报工或产品信息创建抽检工单 |
| 标准管理 | 制定或选择质检标准、抽检规则、判定规则 |
| 任务分配 | 分配质检任务给质检员 |
| 结果审核 | 审核检验结果，确认合格、不合格、返工、复检 |
| 质量追溯 | 查看产品全链路追溯、批次质量统计 |
| 返工协同 | 审核返工单，通知 PMC 或车间安排返工 |

不能做：

| 禁止项 | 说明 |
|---|---|
| 用户授权 | 不能管理用户角色 |
| 生产报工 | 不能代替操作工提交报工 |
| 库存操作 | 不能修改库存和配送状态 |
| 维修派工 | 不能派发设备维修工单 |
| 工艺变更最终生效 | 可以提出质量相关建议，但工艺参数由工艺工程师维护 |

## 21. 首页看板数据可见范围设计

首页看板不是单一权限点。它聚合订单、计划、生产、仓储、质量、设备、追溯、人员和反馈等多类数据，不同角色登录后应看到不同范围、不同颗粒度、不同敏感级别的数据。

看板权限建议拆成两个维度：

| 维度 | 说明 |
|---|---|
| 功能范围 | 角色能看到哪些模块的指标，例如订单、产量、库存、质量、设备、人员 |
| 数据范围 | 角色能看到全厂、部门、车间、产线、仓库、本人相关还是只读演示数据 |

### 21.1 看板信息分类

| 信息类别 | 典型指标 | 敏感等级 |
|---|---|---|
| 订单计划 | 今日订单数、待排产订单、订单完成率、延期风险 | 中 |
| 生产执行 | 今日产量、工单完成率、在制工单、报工数量、停线异常 | 中 |
| 仓储物流 | 库存充足率、缺料预警、领料任务、拣货配送进度 | 中 |
| 质量管理 | 合格率、不良率、返工数、待检任务、质检异常 | 中高 |
| 设备维护 | 设备运行率、故障数、维修中设备、停机时长 | 中 |
| 工艺标准 | SOP 发布状态、工艺参数异常、不良原因统计 | 中 |
| 人员报工 | 员工工时、个人报工、计件工资、班组绩效 | 高 |
| 权限系统 | 用户数量、异常登录、权限申请、审计日志 | 高 |
| 管理反馈 | 未处理反馈、异常闭环率、管理督办事项 | 中 |
| 产品追溯 | 批次追溯、质量链路、工单链路、物料链路 | 中高 |

### 21.2 各角色首页看板可见内容

| 角色 | 可见看板内容 | 数据范围 | 不可见内容 |
|---|---|---|---|
| `SYSTEM_ADMIN` 系统管理员 | 全部业务指标、全部系统指标、权限申请、审计异常、数据库/接口健康状态 | 全厂、全系统 | 明文密码不可见；业务敏感数据可见但必须留审计 |
| `SYSTEM_MAINTAINER` 系统维护员 | 系统运行状态、接口错误、数据库连接、异常登录、权限申请数量、审计日志概览 | 全系统技术视角 | 不看生产经营细节；不看计件工资明细；不直接看质量判定细节 |

| `PMC_PLANNER` PMC计划员 | 订单进度、待排产任务、齐套状态、缺料预警、产线负荷、工单状态、返工待排计划 | 计划部门及相关订单/工单 | 不看用户权限、员工工资明细、设备维修细节、质检内部审核备注 |
| `WORKSHOP_MANAGER` 车间管理员 | 本车间工单进度、产线产量、报工审核待办、异常报工、领料状态、设备故障影响、返工任务 | 本车间、本产线、本班组 | 不看全厂经营汇总；不看其他车间明细；不看用户权限；不看库存成本敏感信息 |
| `PRODUCTION_OPERATOR` 生产操作工 | 本人待接工单、本人已接工单、本人报工状态、本人领料状态、本人设备故障上报、本人报工审核结果 | 本人、本工位、本产线相关 | 不看其他员工报工；不看班组工资汇总；不看用户信息；不看质量主管审核统计；不看全厂数据 |
| `WAREHOUSE_ADMIN` 仓库管理员 | 待处理领料任务、缺料预警、库存充足率、库存异常、拣货任务、配送任务、物料退回任务 | 本仓库、本库区、本物料职责范围 | 不看员工工资；不看质检判定细节；不看生产报工审核细节；不看用户权限 |

| `QUALITY_INSPECTOR` 质检员 | 本人待检任务、本人已检任务、待录入检验项、不合格待处理、相关批次追溯 | 本人质检任务、被分配产品/批次 | 不看质量主管审核总览；不看其他质检员绩效；不看生产工资；不看用户权限 |

| `VIEWER` 只读访客 | 演示用订单、产量、质量、设备、库存概览 | 脱敏演示数据 | 不看用户、工资、审计、权限申请、真实业务敏感明细 |

### 21.3 看板下钻权限

看板上的指标通常可以点击进入详情。详情权限应比概览更严格。

| 看板指标 | 概览可见 | 下钻详情限制 |
|---|---|---|
| 今日产量 | 多数业务角色可见 | 操作工只能看本人/本产线；管理层看汇总；车间管理员看本车间 |
| 工单完成率 | PMC、车间、管理层可见 | PMC 看计划相关工单；车间看本车间；操作工看本人 |
| 缺料预警 | PMC、仓库、管理层可见 | 仓库看物料明细；PMC 看影响订单和计划；管理层看汇总 |
| 报工待审核 | 车间管理员、系统管理员可见 | 操作工只看本人提交结果，不看审核队列 |
| 计件工资 | 操作工、人事、车间、管理层可见不同层级 | 操作工看本人；人事看人员维度；车间看班组汇总；管理层看汇总，不默认看个人明细 |
| 质量合格率 | 质量、管理层、PMC、车间可见 | 质检员看本人任务；质量主管看质量明细；其他角色只看相关工单/批次 |
| 设备运行率 | 设备、车间、管理层可见 | 设备管理员看设备明细；维修员看本人任务；车间看影响本车间生产的设备 |
| 权限申请 | 系统管理员、系统维护员、人事经理可见 | 人事看自己发起/本部门申请；系统维护员看待处理；管理员看全部 |
| 审计异常 | 系统管理员、系统维护员可见 | 普通业务角色不可见 |
| 产品追溯风险 | 质量、PMC、车间、管理层可见 | 按订单、工单、批次、职责范围下钻 |

### 21.4 看板敏感数据处理规则

| 数据 | 处理规则 |
|---|---|
| 用户账号、角色、权限 | 仅系统管理员、系统维护员、人事经理按职责查看 |
| 密码 | 永远不可见，只允许重置，不允许查看 |
| 个人工资/计件明细 | 操作工只看本人；人事按人员职责查看；管理层默认看汇总；访客不可见 |
| 审计日志 | 仅系统管理员、系统维护员可见 |
| 库存成本或高价值物料 | 仓库管理员和管理层按汇总或职责查看，普通操作工不可见 |
| 质量不合格详细原因 | 质量主管、质检员、工艺工程师按职责查看；其他角色只看必要状态 |
| 设备故障详细维修记录 | 设备管理员、设备维护员、车间管理员按职责查看 |
| 跨部门数据 | 默认不可见，除非角色数据范围是 `ALL` 或被授权 |

### 21.5 建议实现方式

后续正式开发时，首页看板接口不建议只有一个固定返回：

```text
GET /api/dashboard/summary
```

建议根据当前登录用户角色和数据范围返回不同内容：

```text
GET /api/dashboard/my-summary
GET /api/dashboard/modules
GET /api/dashboard/production
GET /api/dashboard/quality
GET /api/dashboard/equipment
GET /api/dashboard/warehouse
GET /api/dashboard/system
```

后端根据以下信息过滤：

| 来源 | 用途 |
|---|---|
| `mes_user_role` | 判断用户拥有哪些角色 |
| `mes_role_permission` | 判断是否有某类看板权限 |
| `mes_role_data_scope` | 判断可见全厂、部门、产线、仓库还是本人 |
| 业务表中的 `created_by`、`assigned_to`、`accepted_by`、`operator_id`、`inspector_id` 等字段 | 判断本人相关数据 |

## 22. 后续正式设计建议

1. 建议从单字段 `role_code` 逐步升级为 `mes_role`、`mes_permission`、`mes_role_permission`、`mes_user_role` 表。
2. 增加 `mes_permission_apply` 权限变更申请表，支持人事经理申请、系统维护员处理、系统管理员复核。
3. 增加 `mes_audit_log` 审计日志表，记录登录、密码重置、角色变更、删除、审批、库存调整、质检判定、维修关闭等关键动作。
4. 登录 token 应从前端本地演示 token 升级为后端签发、可过期、可校验的会话或 JWT。
5. 机器人和外部系统使用独立 API 凭证，不与人员角色混用。
6. 权限控制需要同时考虑“角色权限”和“数据范围”，例如操作工只能看本人/本产线工单，总经理可以看全厂汇总。
7. 正式实现前，应先由你审核本角色清单，确认是否需要保留“生产主管”和“仓储主管”这类中间管理角色。
