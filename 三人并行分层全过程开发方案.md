## 1. 给所有 Codex 的总指令

如果你是组员交给 Codex 的执行者，请先阅读本节，再阅读自己负责的“开发者 A/B/C”章节。

### 1.1 目标

在现有项目基础上开发一个按以下层次组织的 MES 系统：

```text
UI 层
  ↓
控制层 Controller / Resource
  ↓
业务层 Service
  ↓
DAO 层
  ↓
数据库 MySQL
```

系统围绕报告中的业务主线开发：

```text
订单下达
→ 生产排产
→ 齐套分析 / 欠料预警
→ 生产工单创建 / 接收 / 派发
→ 领料任务
→ 仓储备货 / 拣货
→ 机器人配送
→ 生产报工 / 计件
→ 质量抽检 / 结果判定
→ 返工 / 设备报修 / 维修维护
→ 产品追溯
→ 综合生产看板 / 管理反馈
```

### 1.2 主工程选择

统一使用 `MES System` 目录作为 Web 主工程。

原因：

- `MES System` 已经是 Maven WAR 工程。
- 已有 Jakarta REST 入口：`HelloApplication.java`，路径为 `/api`。
- 适合放置 REST 控制层和静态 UI。

根目录下的 `src/main/java/com/mes/jdbc` 仅作为 JDBC 连接和测试写法参考，不作为最终主工程继续扩展。

### 1.3 技术栈约定

```text
语言：Java 21
构建：Maven
Web：Jakarta REST / Jakarta Servlet
数据库：MySQL
数据访问：原生 JDBC，使用 PreparedStatement
测试：JUnit 5
UI：静态 HTML + CSS + JavaScript fetch 调用 REST API
返回格式：统一 JSON
```

开发时需要在 `MES System/pom.xml` 中补充 MySQL 驱动依赖：

```xml
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
    <version>8.4.0</version>
</dependency>
```

如需要 JSON 序列化，优先使用运行容器自带 Jakarta JSON 支持；若项目本地运行不便，可补充 Jackson：

```xml
<dependency>
    <groupId>com.fasterxml.jackson.core</groupId>
    <artifactId>jackson-databind</artifactId>
    <version>2.17.2</version>
</dependency>
```

### 1.5 分层职责

#### UI 层

职责：

- 展示数据列表、表单、状态按钮和看板指标。
- 使用 `fetch` 调用 REST API。
- 不直接拼 SQL。
- 不写业务规则。
- 只做简单输入校验，例如必填、数字非负。

每个开发者负责自己模块的 UI 区块，可以放在同一个 `index.html` 中，也可以拆成多个页面。

#### 控制层 Resource

职责：

- 接收 HTTP 请求。
- 解析路径参数、查询参数和 JSON 请求体。
- 调用 Service。
- 返回统一 `ApiResponse<T>`。
- 不直接访问数据库。
- 不写复杂业务判断。

命名规则：

```text
XxxResource.java
```

示例：

```text
ProductionTaskResource
WorkOrderResource
InventoryResource
QualityInspectionResource
DashboardResource
```

#### 业务层 Service

职责：

- 实现业务流程和状态流转。
- 校验业务前置条件。
- 调用一个或多个 DAO。
- 生成业务编号。
- 控制跨表写入顺序。
- 必要时开启事务。

命名规则：

```text
XxxService.java
```

示例：

```text
ProductionTaskService.releaseTask()
WorkOrderService.dispatchWorkOrder()
MaterialRequisitionService.createFromWorkOrder()
WorkReportService.submitReport()
QualityInspectionService.judgeInspection()
MaintenanceOrderService.createFromRepairReport()
```

#### DAO 层

职责：

- 只负责 SQL。
- 使用 `PreparedStatement` 防止 SQL 注入。
- 一张核心表至少一个 DAO。
- 提供基础 CRUD 和必要查询。
- 不写业务状态流转。

命名规则：

```text
XxxDao.java
```

示例：

```text
CustomerOrderDao
ProductionTaskDao
WorkOrderDao
InventoryDao
WorkReportDao
QualityInspectionDao
EquipmentDao
DashboardMetricDao
```

#### 数据库层

职责：

- 使用 `database_design_output/mes_table_metadata.json` 作为字段权威来源。
- `database_design_output/mes_schema_mysql.sql` 当前存在表名占位异常：`CREATE TABLE $(System.Collections.Specialized.OrderedDictionary.name)`，开发前必须修复或重新生成可执行 SQL。
- 建议数据库名：`MES`。
- 表名采用 `mes_` 前缀。
- 字段名保持元数据中的蛇形命名。

### 3.2 数据库修复任务

当前 `database_design_output/mes_schema_mysql.sql` 的 `CREATE TABLE` 表名是异常占位文本。开发前必须执行以下任一方案：

方案 A：根据 `mes_table_metadata.json` 重新生成建表 SQL。

方案 B：手工修复 `mes_schema_mysql.sql` 中每个 `CREATE TABLE` 的表名。

优先建议方案 A。生成规则：

```text
1. 读取 mes_table_metadata.json。
2. 每个对象的 name 是表名。
3. fields 数组生成字段。
4. pk=true 的字段生成 PRIMARY KEY。
5. constraint 中包含 UK 的字段生成 UNIQUE KEY。
6. constraint 中包含 IDX 的字段生成普通索引。
7. constraint 中包含 FK -> xxx.yyy 的字段生成外键，若时间不够，第一版可先不加外键，只保留字段和索引。
```

第一版必须保证：

```text
1. 所有表可成功创建。
2. 所有主键自增。
3. 所有业务编号字段可唯一。
4. 种子数据可插入。
5. 三人负责的模块能使用相同数据库。
```

### 3.3 数据库初始化入口

建议创建：

```text
MES System/src/main/java/com/example/messystem/common/DatabaseInitializer.java
```

职责：

```text
1. 创建 MES 数据库。
2. 执行修复后的 schema SQL。
3. 执行 mes_seed_data.sql。
4. 可重复执行，不重复插入唯一数据。
```

如果时间紧，可以不做自动初始化，由开发者手动执行 SQL，但必须在 README 中写清楚执行顺序。

## 6. 业务层全过程方案

业务层按报告中的流程闭环实现。

### 6.1 生产计划闭环

```text
创建订单
→ 创建生产任务
→ 齐套分析
→ 若物料不足，生成欠料预警
→ 若齐套通过，发布生产任务
→ 创建生产工单
→ 派发生产工单
```

核心 Service：

```text
CustomerOrderService
ProductionTaskService
KittingService
WorkOrderService
```

### 6.2 仓储物流闭环

```text
根据工单创建领料任务
→ 仓库审核领料
→ 拣货
→ 生成机器人配送任务
→ 机器人送达
→ 生产操作工确认收料
→ 扣减库存并记录库存流水
```

核心 Service：

```text
InventoryService
MaterialRequisitionService
PickingTaskService
RobotDeliveryService
```

### 6.3 生产报工闭环

```text
操作工提交报工
→ 系统校验工单状态
→ 车间管理员审核
→ 更新工单实际数量
→ 生成计件工资
→ 报工数据提供给质量模块
```

核心 Service：

```text
WorkReportService
PieceworkWageService
```

### 6.4 质量闭环

```text
根据报工或工单创建质检单
→ 录入质检项目
→ 判定合格 / 不合格
→ 不合格生成返工单
→ 写入质量追溯
```

核心 Service：

```text
QualityInspectionService
ReworkOrderService
QualityTraceService
```

### 6.5 设备维护闭环

```text
操作工上报设备故障
→ 设备状态改为 FAULT
→ 管理员审核报修
→ 生成维修工单
→ 维护员完成维修
→ 设备状态恢复 IDLE / RUNNING
```

核心 Service：

```text
EquipmentService
EquipmentRepairReportService
MaintenanceOrderService
MaintenancePlanService
```

### 6.6 追溯看板闭环

```text
聚合订单、任务、工单、领料、报工、质检、返工、设备数据
→ 生成产品追溯记录
→ 生成看板指标

```

核心 Service：

```text
ProductTraceService
```

## 7. DAO 层全过程方案

### 7.1 DAO 基本方法

每个 DAO 至少实现：

```java

List<Entity> findAll(int page, int pageSize) throws SQLException;
```

### 7.2 事务处理

涉及多表写入的 Service 必须使用同一个 `Connection`：

```text
1. 创建工单 + 写工单操作日志
2. 齐套分析 + 缺口明细 + 欠料预警
3. 领料审核 + 拣货任务 + 库存流水
4. 报工审核 + 更新工单实际数量 + 生成计件工资
5. 质检判定 + 返工单 + 质量追溯
```

## 17. 风险和取舍

### 17.1 时间不够时优先级

必须完成：

```text
1. 数据库可用。
2. 主流程可跑通。
3. 三人接口能联调。
4. UI 能演示关键操作。
```

可以简化：

```text
1. 权限只做 role_code 展示，不做登录拦截。
2. 外部 ERP / WMS / 财务系统只记录同步日志，不做真实集成。
3. 机器人只模拟状态流转，不做真实路径规划。
4. 看板指标可实时 SQL 聚合，不一定落库 mes_dashboard_metric。
5. 外键可先不强制，业务层保证关联。
```

不能省略：

```text
1. 分层结构。
2. PreparedStatement。
3. 统一 API 返回格式。
4. 工单、领料、报工、质检、设备维修的状态流转。
5. 产品追溯链路。
```

### 17.2 当前项目特别注意

```text
1. 根目录 README 目前只是 JDBC Demo，不代表最终系统。
2. MES System 目前只有 HelloResource，需要扩展成真实 REST API。
3. database_design_output/java_entities 中实体没有 package、getter、setter，需要整理后再使用。
4. mes_schema_mysql.sql 当前表名异常，不能直接作为最终建表脚本。
5. 三人开发时必须避免分别维护三套 DbConfig。
```

## 18. 一句话总结

本项目按“UI 层、控制层、业务层、DAO 层、数据库层”开发；三人分别负责“计划工单”“仓储生产”“质量设备看板”三条业务线；通过统一数据库、统一接口、统一状态和统一返回格式并行推进，最后用一条“订单到追溯看板”的完整流程进行联调和演示。
