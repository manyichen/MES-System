# MES 新旧版本融合说明与 Codex 执行提示词

## 1. 文档用途

本文档用于指导组员将自己的本地修改与当前新版 MES 项目进行融合，尤其适用于以下场景：

1. 组员本地仍在使用旧静态前端、旧目录结构和旧接口。
2. 组员已经在旧代码上继续修改后端接口、业务流程或权限管理。
3. 新版代码已经改为 Vue 前端，并对后端接口、目录结构和业务分层进行了大范围重构。
4. 组员通过 Git 暂存本地修改、拉取新版代码、恢复暂存后产生大量冲突。
5. 组员使用 Codex 分析和解决冲突，而不是人工逐行合并。

本文档既是人工融合说明，也是可以直接复制给 Codex 的任务提示词。

---

## 2. 最终融合目标

融合完成后的代码必须同时满足以下目标：

- 前端整体框架、目录结构、路由、状态管理和请求方式使用当前新版 Vue 实现。
- 不保留旧版静态 HTML、全局 JavaScript、手工 DOM 拼装和重复 CSS 框架。
- 后端必须采用新版的 `controller`、`service`、`dao`、`entity` 分层结构。
- 对外接口的方法、路径、请求结构、响应结构和异常格式优先与新版规范接口一致。
- 同一功能模块中存在新旧重复接口时，最终只能保留一个规范接口。
- 组员最新代码中的有效业务修改必须保留，不能因为接受新版文件而整体丢失。
- 权限管理的实际业务语义以组员代码中“最新有效接口”的实现为主要参考。
- 组员旧代码中的旧接口、旁路接口、失效权限分支和重复实现不能直接搬入新版。
- 权限逻辑必须迁移到新版规范接口及新版分层结构中，而不是恢复旧接口。
- 前端权限显示与后端权限校验必须使用同一套权限编码和角色语义。
- 未能确认权限语义的接口不得默认向所有角色开放，应记录为待确认项并保持默认拒绝。

最终效果应当是：

> 代码结构和接口形式与当前新版一致；组员最新开发成果得到保留；接口内部的角色权限、数据范围、用户归属、职责分离和状态限制，使用组员已经实现并验证过的最新逻辑。

## 3. 融合优先级和事实来源

Codex 必须按照下表判断冲突双方谁是主要事实来源，不能简单选择 `ours` 或 `theirs`。

| 内容 | 主要事实来源 | 融合原则 |
|---|---|---|
| 前端技术框架 | 当前新版 | 必须使用 Vue 3、Vite、Pinia、Vue Router 和统一 API 层 |
| 前端页面布局与导航框架 | 当前新版 | 旧页面只能提供业务字段、操作流程和权限显示参考 |
| 前端接口调用证据 | 双方对照 | 新版决定规范路径，组员旧前端可用于识别其最新业务修改 |
| 后端目录和分层 | 当前新版 | Controller 不执行 SQL，Service 不直接使用 JDBC，DAO 不依赖 HTTP 类型 |
| 规范接口方法和路径 | 当前新版优先 | 组员确有新版新增接口且新版分支缺失时，按新版命名规范补入 |
| 接口请求和响应格式 | 当前新版优先 | 保留组员新增字段，但统一使用新版响应包和异常处理 |
| 权限编码和角色约束 | 组员最新有效实现 | 将权限语义迁移到新版接口，不恢复旧接口 |
| 数据范围和数据归属 | 组员最新有效实现 | 迁移产线、仓库、本人数据、工单归属等校验 |
| 状态机和业务校验 | 对比双方最新修改 | 保留限制更完整、证据更充分且符合正式流程的一方 |
| SQL、表字段和数据库迁移 | 双方合并后验证 | 不删除组员新增字段；迁移脚本必须可重复执行并保持兼容 |
| 重复接口 | 当前新版规范接口 | 提取旧接口中的有效权限/业务逻辑后删除旧接口 |
| 测试 | 双方有效测试合并 | 测试必须改为新版接口和目录，不允许继续调用已删除接口 |

### 3.1 关于“新版权限管理”的特别说明

当前新版代码中可能已经存在以下类或概念：

- `AuthFilter`
- `AuthorizationPolicy`
- `AuthenticatedUser`
- `DataScopeFilter`
- `DataScopeService`
- 角色、权限、用户角色和数据范围 DAO
- Vue 权限导航和按钮显示

这些文件的存在不能证明实际权限行为已经正确。用户已经明确指出当前版本存在“不同角色实际权限表现相同或权限边界不符合要求”的问题。

因此 Codex 必须：

1. 按接口验证真实授权行为，不能只检查是否存在权限类。
2. 读取组员冲突代码中的最新权限校验。
3. 将组员已经实现的权限语义迁移到新版规范接口。
4. 不要仅把旧权限类文件覆盖到新版，也不要仅保留新版空壳权限代码。
5. 不要自行发明一整套与组员实现无关的权限体系。
6. 对组员代码没有覆盖的权限缺口形成清单，默认拒绝并交由后续统一完善。

## 4. Git 暂存、拉取和恢复建议

### 4.1 组员操作前必须先备份

在拉取新版前，组员应先确认当前分支和工作区：

```powershell
git status
git branch --show-current
git diff --stat
```

建议创建一个只用于保留现场的备份分支：

```powershell
git branch backup/before-vue-merge-日期
```

然后暂存全部已跟踪和未跟踪文件：

```powershell
git stash push -u -m "before-vue-backend-refactor-merge"
```

拉取新版后，优先使用：

```powershell
git stash apply stash@{0}
```

不建议一开始使用 `git stash pop`。`apply` 会保留暂存记录，即使融合过程出错仍可重新恢复。全部验证通过后再执行：

```powershell
git stash drop stash@{0}
```

### 4.2 Codex 处理冲突时的 Git 禁止事项

Codex 不得执行以下操作：

- 不得使用 `git reset --hard`。
- 不得对整个项目执行 `git checkout --ours .`。
- 不得对整个项目执行 `git checkout --theirs .`。
- 不得直接删除组员全部冲突文件。
- 不得因为新版能编译就认定组员修改无用。
- 不得在未核对权限逻辑前删除旧接口文件。
- 不得在未验证前删除 stash。
- 不得自动提交或推送，除非组员明确要求。

`stash apply` 产生冲突时，一般可以把当前拉取后的新版理解为结构基线，把 stash 中的代码理解为组员修改来源，但 Codex 仍必须逐文件分析，不能机械接受某一侧。

## 5. Codex 开始融合前必须完成的扫描

Codex 在修改代码前必须先读取项目，不得直接开始解决冲突标记。

### 5.1 Git 和冲突状态

至少执行：

```powershell
git status --short
git diff --name-only --diff-filter=U
git stash list
```

输出并分类：

- Vue 前端冲突文件
- 旧静态前端文件
- Controller 冲突
- Service 冲突
- DAO 冲突
- Entity/DTO 冲突
- SQL/数据库迁移冲突
- 权限和角色相关冲突
- 测试冲突
- 文档冲突

### 5.2 接口扫描

分别扫描双方代码中的：

- `@Path`
- `@GET`
- `@POST`
- `@PUT`
- `@DELETE`
- `@PATCH`
- 前端 `/api/` 调用
- 权限编码字符串
- 角色编码字符串
- Service 调用链
- DAO SQL 表和字段

不得只按接口路径判断是否重复，还要同时比较：

- HTTP 方法
- 请求参数
- 请求体字段
- 响应字段
- 权限要求
- 角色范围
- 数据范围
- 用户归属
- 状态前置条件
- 状态变更结果
- 审计和通知副作用

### 5.3 建立接口映射表

在修改代码前建立一份临时接口映射表，至少包含：

| 模块 | 组员旧/新接口 | 新版规范接口 | 是否仍被调用 | 最新权限来源 | 数据范围 | 状态限制 | 融合决定 |
|---|---|---|---|---|---|---|---|
| 示例：工单派发 | `POST /old/work-order/dispatch` | `POST /work-orders/{id}/dispatch` | 是 | 组员最新 Controller/Service | 指定产线 | 仅 CREATED | 迁移逻辑后删除旧接口 |

接口映射表必须覆盖全部冲突接口，不能只处理编译报错接口。

## 6. 如何识别组员代码中的“最新有效接口”

组员旧代码可能在同一模块中同时存在早期接口和后来新增接口。Codex 不得把所有接口都保留，也不得仅按文件修改时间判断。

应按照以下证据顺序识别最新有效接口：

1. 组员最新前端实际调用的接口。
2. 组员最新测试覆盖的接口。
3. 最新 Service 调用链正在使用的接口。
4. 最新数据库字段和迁移脚本对应的接口。
5. 权限说明、角色文档和验收账号正在描述的接口。
6. Git 历史中后提交、且明确替代旧实现的接口。
7. 状态机更完整、数据范围更严格、旁路更少的实现。

以下情况通常代表旧接口：

- 路径中带有 `legacy`、`old`、`temp`、`demo` 等临时语义。
- 新旧两个接口完成完全相同的 CRUD。
- 旧接口直接操作数据库，新接口已经调用 Service。
- 旧接口绕过审核、派工、接收、验收等正式状态机。
- 旧接口没有被任何前端、测试或 Service 使用。
- 旧接口权限更宽松，仅靠前端隐藏按钮。
- 旧接口使用已经废弃的角色编码。

如果仍无法判断，Codex 必须保留证据更充分的一套实现，并在最终报告中列出待确认项，不能同时暴露两个功能相同的接口。

## 7. 前端融合规则

### 7.1 必须保留的新版框架

融合后的前端必须继续使用：

- Vue 3
- Vite
- Pinia
- Vue Router
- `frontend/src/api/` 统一请求层
- `frontend/src/stores/` 会话和权限状态
- `frontend/src/config/` 模块与业务动作配置
- `frontend/src/components/` 公共组件
- `frontend/src/views/` 页面视图
- Lucide 图标

旧版以下内容不得恢复为运行时主框架：

- `frontend/js/*.js` 全局脚本
- 手工 `document.querySelector` 页面拼装
- HTML 中的大段内联事件
- 旧版静态登录页和静态模块页
- 旧版 `trace-public.html`
- 重复的全局 CSS
- 每个模块各自实现一套请求和令牌逻辑

### 7.2 旧前端允许迁移的内容

旧前端只能作为以下信息来源：

- 组员新增的表格字段
- 新增的业务按钮
- 最新接口路径证据
- 权限按钮显示条件
- 角色可见模块
- 表单字段和校验提示
- 状态对应的可操作动作
- 组员新增的业务流程

这些内容必须重写为 Vue 组件、模块配置、组合式逻辑或统一 API 调用，不能直接把旧 JavaScript 粘贴回项目。

### 7.3 前端权限融合

前端权限控制只能作为用户体验层，后端仍是最终权限边界。

Codex 需要将组员旧前端中的权限显示条件转换为新版形式，例如：

- 导航项使用 `roles` 或 `permissions` 配置。
- 按钮使用权限编码控制显示。
- 行操作同时检查权限和业务状态。
- 个人范围、产线范围和仓库范围不在前端自行伪造过滤结果。
- 接口返回 401 时清理会话并跳转登录。
- 接口返回 403 时显示无权访问，不得自动改用旧接口绕过。

如果组员旧前端使用的是角色名称判断，而后端已经使用权限编码，应保留其业务意图，并转换为新版统一权限编码。

## 8. 后端融合规则

### 8.1 Controller 层

Controller 只负责：

- 接收路径、查询参数和请求体
- 获取当前认证用户
- 调用 Service
- 返回统一响应结构
- 将异常交给统一异常处理

Controller 不得：

- 直接调用 DAO
- 直接获取数据库连接
- 编写 SQL
- 在多个 Controller 中复制同一权限规则
- 绕过 Service 修改状态

如果组员最新权限逻辑写在旧 Controller 中，Codex 必须提取其业务语义，再放入新版授权策略、数据范围 Service 或业务 Service，不能原样保留旧分层。

### 8.2 Service 层

Service 负责：

- 角色和权限业务语义
- 用户归属校验
- 产线、仓库和本人数据范围
- 状态机判断
- 请求参数业务校验
- 多 DAO 用例编排
- 事务用例的业务边界
- 审计、通知和关联业务动作

Service 不得直接使用：

- `Db.getConnection()`
- `Connection`
- `PreparedStatement`
- `ResultSet`
- SQL 字符串

组员旧 Service 中的权限判断应尽量保留，但必须适配新版实体、DAO 和异常结构。

### 8.3 DAO 层

DAO 负责：

- SQL 查询和更新
- 结果映射
- 数据是否存在
- 数据归属查询
- 必须原子执行的持久化事务
- 比较并更新式的状态变更

DAO 不得依赖：

- JAX-RS Response
- HTTP Header
- Servlet Request
- Vue 或前端字段显示逻辑

组员旧 Controller/Service 中的 SQL 必须下沉到 DAO。下沉时不得把完整业务流程全部塞入 DAO。

### 8.4 Entity/DTO

融合请求和响应字段时：

- 新版字段命名为主要规范。
- 组员新增且仍被最新接口使用的字段必须保留。
- 同义字段只保留一个规范名称。
- 确需兼容旧前端时，兼容应是临时 DTO 映射，不得长期暴露两套接口。
- 不得将密码哈希、令牌哈希或内部审计字段返回前端。

## 9. 权限管理逻辑的融合要求

这是本次融合最重要的部分。

### 9.1 必须识别的权限维度

Codex 对每个接口至少检查以下维度：

1. 是否必须登录。
2. 需要哪个权限编码。
3. 哪些角色能够拥有该权限。
4. 是否限制本人数据。
5. 是否限制指定产线。
6. 是否限制指定仓库。
7. 是否限制工单的被派工人或接收人。
8. 是否禁止申请人审核自己的申请。
9. 是否禁止执行人验收自己的结果。
10. 是否要求特定业务状态。
11. 是否要求上游单据已经审核。
12. 是否需要记录审计日志或通知。

### 9.2 权限语义迁移示例

如果组员旧代码中存在：

```text
旧接口 A：生产操作工只能查看 assigned_to/accepted_by 为本人的工单
新版接口 B：GET /work-orders
```

正确融合方式：

- 保留新版 `GET /work-orders`。
- 将旧接口 A 中的本人归属逻辑迁入新版 WorkOrder Service/DAO。
- Controller 根据认证用户决定使用全量查询还是操作工范围查询。
- 删除旧接口 A。
- Vue 继续调用新版 `GET /work-orders`。

错误融合方式：

- 同时保留 A 和 B。
- 让操作工前端调用 A，其他角色调用 B。
- 只在 Vue 隐藏数据，后端 B 仍返回全部工单。

### 9.3 不得丢失的常见权限逻辑

需要重点搜索并迁移：

- `hasRole`
- `hasPermission`
- 角色编码比较
- 权限编码比较
- `currentUser.userId`
- `assignedTo`
- `acceptedBy`
- `requestedBy`
- `createdBy`
- `maintainerId`
- `inspectorId`
- 用户产线范围
- 用户仓库范围
- 申请人和审核人不能相同
- 执行人和验收人不能相同
- 只允许当前状态执行动作
- 数据详情访问前的归属查询

Codex 不能只迁移 `hasPermission` 这一行，而丢掉其后的数据范围和状态条件。

### 9.4 当前权限缺口的处理

如果新版和组员代码都没有明确某个接口的权限逻辑：

- 不要默认所有角色可用。
- 不要根据前端菜单推断后端权限。
- 保持认证要求和默认拒绝。
- 将接口、建议权限编码、可能角色和缺失证据列入最终待办。
- 等用户后续统一设计权限时再开放。

## 10. 各模块重点融合检查

### 10.1 系统与账号

- 登录、退出、当前用户
- 用户创建和资料修改
- 角色分配
- 权限申请、审核和执行
- 数据范围分配
- 会话撤销
- 审计日志
- 系统健康检查

重点检查系统管理员和人事经理的职责是否混淆。

### 10.2 计划与工单

- 客户订单
- 生产任务
- 齐套分析
- 缺料预警
- 工单创建、派发、接收和拒绝
- 工单操作日志

重点保留组员最新的派工角色限制、操作工本人归属和车间产线范围。

### 10.3 生产报工

- 操作工创建报工
- 车间管理员审核
- 重复审核保护
- 实际产量更新
- 计件工资生成
- 本人工资范围

### 10.4 仓储物流

- 领料申请
- 仓库接收和审批
- 库存校验
- 拣货
- 配送
- 收料确认
- 外部采购入库
- 库存流水

重点保留操作工工单归属、仓库数据范围和重复收料保护。

### 10.5 质量管理

- 质检任务创建和分配
- 质检员录入与提交
- 质量主管审核
- 合格、拒绝和返工状态
- 返工闭环
- 质量追溯

重点避免质检员审核自己的结果。

### 10.6 设备管理

- 设备台账
- 故障上报
- 报修审核
- 转维修单
- 维修派工
- 维护员执行

- 产线

## 11. 数据库融合规则

Codex 必须对比：

- 新版数据库脚本
- 组员新增迁移脚本
- Entity 字段
- DAO SQL 字段
- 前端表单字段

处理原则：

- 保留组员最新业务需要的新增字段。
- 同一个字段只保留一个正式定义。
- 迁移脚本尽量使用 `IF NOT EXISTS`、`ON CONFLICT` 等可重复执行形式。
- 不在代码启动时反复插入测试业务数据。
- 不自动执行数据清理脚本。
- 不自动连接云端数据库执行破坏性测试。
- 不提交 `.env`、数据库密码、令牌或清理前备份。

当前项目写库集成测试连接远程数据库时默认禁用。Codex 不得为了让测试数量变多而移除该保护。确需执行时只能使用独立测试数据库，并显式设置：

```powershell
-Dmes.allowDestructiveRemoteDbTests=true
```

## 12. 冲突处理顺序

Codex 应按以下顺序解决冲突：

1. 保留备份和 stash，不做破坏性 Git 操作。
2. 解决构建文件和目录结构冲突。
3. 确立 Vue 前端为唯一前端框架。
4. 确立新版 Controller/Service/DAO 包结构。
5. 建立完整接口映射表。
6. 识别组员每个模块的最新接口。
7. 提取这些接口中的权限、数据范围和状态机逻辑。
8. 将逻辑迁入新版规范接口和分层结构。
9. 修改 Vue 调用与字段映射。
10. 合并数据库迁移和实体字段。
11. 删除已完成迁移的旧接口和旧静态前端。
12. 修复测试调用。
13. 执行接口重复扫描、构建和角色验收。
14. 最后才删除 stash。

不要先删除旧代码再试图回忆其中的权限逻辑。

## 15. 可直接复制给 Codex 的完整提示词

下面内容可以直接复制给组员所使用的 Codex：

```text
你现在需要完成一次 MES 项目的新旧版本深度融合。不要把任务理解为简单解决 Git 冲突，也不要机械选择 ours 或 theirs。你需要读懂双方代码、识别组员最新业务修改，并将其融合到当前新版架构中。

一、背景

1. 当前拉取后的新版已经把前端从旧静态 HTML/全局 JavaScript 改为 Vue 3 + Vite + Pinia + Vue Router。
2. 新版后端已经按照 controller、service、dao、entity 分层，并删除了一批重复或旁路接口。
3. 组员本地 stash 中仍包含旧前端、旧目录、旧接口，以及组员后续新增的接口、业务逻辑和权限管理。
4. 同一功能模块可能同时存在早期旧接口和后来新增接口。你必须识别组员代码中的最新有效接口，不能把全部旧接口恢复。
5. 用户明确指出当前新版实际权限表现仍可能不正确。即使看到 AuthFilter、AuthorizationPolicy、DataScopeService 或前端权限菜单，也不能直接认定权限已经完成。

二、不可改变的总体原则

1. 前端技术框架和整体目录必须以当前新版 Vue 为准。
2. 不得恢复 frontend/js 下的旧全局脚本、静态模块页、手工 DOM 拼装或旧 trace-public.html。
3. 后端分层必须以当前新版为准：Controller 不执行 SQL，Service 不直接使用 JDBC，DAO 不依赖 HTTP 类型。
4. 对外接口的方法、路径、请求结构、响应包和异常格式优先与当前新版规范接口一致。
5. 权限管理的业务语义以组员代码中“最新有效接口”的实现为主要来源。
6. 必须把旧接口中的有效权限逻辑迁移到新版规范接口，而不是保留两套接口。
7. 组员最新新增的业务字段、状态限制、数据范围、用户归属、职责分离、审计和通知逻辑不能丢失。
8. 无法确认权限语义的接口不得默认向所有角色开放，应保持默认拒绝并列入待确认清单。
9. 不得运行 git reset --hard，不得整项目 checkout --ours/--theirs，不得删除 stash，不得自动推送。

三、开始前的强制扫描

1. 执行 git status --short、git diff --name-only --diff-filter=U、git stash list。
2. 列出所有冲突文件，并按 Vue、旧前端、Controller、Service、DAO、Entity、SQL、权限、测试、文档分类。
3. 扫描所有 JAX-RS 接口、前端 API 调用、权限编码、角色编码、数据范围和状态校验。
4. 在修改前建立接口映射表，记录组员接口、新版规范接口、调用证据、最新权限来源、数据范围、状态机和融合决定。
5. 不能只处理当前编译错误；必须扫描整个项目。

四、识别最新接口

识别组员最新有效接口时，优先参考：最新前端调用、最新测试、Service 调用链、数据库迁移、权限文档、Git 历史和完整状态机。不要按文件出现顺序判断。

如果两个接口功能重复：保留新版规范路径；提取组员最新接口中的权限和业务逻辑；迁入新版 Service/DAO；更新 Vue；确认无消费者后删除旧接口。

五、权限融合

对每个接口逐项检查：登录要求、权限编码、允许角色、本人数据、产线范围、仓库范围、工单归属、申请与审核职责分离、执行与验收职责分离、状态前置条件、上游审核条件、审计和通知。

重点搜索 hasRole、hasPermission、currentUser、assignedTo、acceptedBy、requestedBy、createdBy、maintainerId、inspectorId、line scope、warehouse scope 以及状态判断。

不要只迁移权限判断的一行代码，还必须迁移关联的数据范围、归属和状态限制。

如果旧权限逻辑在 Controller 中，将其业务语义迁入新版授权策略、DataScopeService 或业务 Service；SQL 下沉 DAO。不要为了保留权限而破坏新版分层。

六、前端融合

保留新版 Vue 框架。旧前端只作为字段、操作、最新接口和权限显示的证据。将组员新增功能改写为 Vue 配置、组件、Store 或统一 API 调用，不得直接粘贴旧脚本。

前端权限只是显示层，后端必须实际拒绝越权请求。401 清理会话，403 显示无权访问，不能自动改用旧接口绕过。

七、数据库和测试

合并双方新增字段和迁移脚本，确保 Entity、DAO、前端字段一致。不要提交数据库密码、令牌、.env 或清理前备份。

当前写库集成测试在远程数据库上默认禁用，不得移除保护。不要对云端数据库运行破坏性测试。确需执行只能使用独立测试数据库。

八、验证

1. 确认无 Git 冲突标记。
2. 确认无重复 HTTP 方法和规范化路径。
3. 运行前端 test、build 和浏览器 smoke。
4. 运行后端 test 和 package。
5. 验证 12 个正式角色，每个角色至少测试一个允许接口和一个禁止接口。
6. 验证本人、产线、仓库数据范围以及非法状态操作。
7. 检查 Vue 的全部接口调用都有后端实现，后端保留接口都有明确消费者或业务用途。

九、最终报告

完成后报告：冲突解决文件、删除的旧代码、保留的组员修改、接口映射、权限迁移、数据范围、数据库变化、测试结果、权限缺口和 stash 状态。

在任务真正完成前不要只给方案。请持续执行扫描、修改和验证。遇到无法确定的新旧接口或权限语义时，先保留证据更充分且限制更严格的实现，记录问题，不要同时开放两套接口。
```

## 16. 最终验收结论标准

只有同时满足以下条件，才可以认为融合完成：

- Vue 是唯一正式前端入口。
- 旧静态前端不再参与运行。
- 后端分层测试通过。
- 没有重复接口。
- 没有未解决 Git 冲突标记。
- 组员最新业务修改已经迁入新版结构。
- 组员最新权限逻辑已经迁入新版规范接口。
- 新版接口没有因为融合重新变成所有角色都可访问。
- 12 个正式角色的允许和拒绝场景均经过验证。
- 云端数据库没有因测试再次产生大量调试数据。
- stash 在完整验证前仍然保留。
